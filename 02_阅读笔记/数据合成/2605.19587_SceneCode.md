# 📖 花火格式笔记

## 🎯 题目

SceneCode: Executable World Programs for Editable Indoor Scenes with Articulated Objects

## 📝 三句摘要

1. **问题背景**：室内场景生成多输出static mesh，无法支持下游机器人的"门能不能开/抽屉关节在哪/碰撞体是否干净"等可交互性查询，场景也不可编辑和批量变体生成。
2. **核心方法**：将自然语言prompt编译为可执行的part-wise Blender Python program，导出SDF给物理仿真器使用；同时维护scene-state registry，将object request、可执行程序、渲染几何、仿真资产关联起来。
3. **关键结果**：生成的场景代码可编辑、可复用、可批量生成变体，并支持物理仿真器的关节/碰撞/抓取查询。

## 💎 价值评估

- **🔬 研究价值**：路线C（程序化世界代码）的代表，提出"场景即代码"的范式，对主人设计低空SceneSpec/TaskSpec中间格式有直接启发。
- **🚀 实践价值**：主人不应只保存UE level，最好保存SceneSpec JSON——建筑类型、道路网、目标区域、天气参数等，让场景可复现、可编辑、可批量变体生成，而非一次性UE资产。
- **📈 扩展潜力**：可扩展到低空TaskSpec设计（巡检路线/覆盖区域/避障约束），以及AnnotationSpec（语义标注规格），形成完整的可执行规范体系。

## 🎯 可落地实验点

**实验设计**：参考SceneCode的scene-state registry，为低空场景设计SceneSpec Schema——包含global_layout（道路/建筑/植被/开放区域）、semantic_objects（带bbox_3d和category）、task_zones、collision_layers、weather_lighting字段，支持Python程序读取并驱动UE PCG生成。
- 对比基线：纯UE工程文件（不可程序化修改）、纯JSON Spec（无执行逻辑）
- 度量指标：Spec可执行率、场景生成可复现性、变体生成速度
- 预期结果：SceneSpec Schema使场景可复现性从UE工程文件的~50%提升到~95%以上
