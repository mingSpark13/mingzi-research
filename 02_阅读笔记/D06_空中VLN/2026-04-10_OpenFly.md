# OpenFly（摘要级速记）

- **日期**: 2026-04-10
- **arXiv**: 2502.18041
- **标题**: OpenFly: A Comprehensive Platform for Aerial Vision-Language Navigation
- **方向**: D06 空中视觉语言导航
- **阶段**: 摘要级深挖

## 核心贡献
OpenFly 不只是又做了一套 aerial VLN 数据，而是把 **UE / GTA V / Google Earth / 3D Gaussian Splatting** 四类渲染源、自动轨迹生成、自动指令生成、统一接口和 benchmark 一起打包成完整平台。论文同时提出 **OpenFly-Agent**，用 **keyframe selection + visual token merging** 做轻量历史压缩，在 seen / unseen 场景上都明显优于对比方法。

## 与本方向的关系
它更像 D06 的 **数据工厂 + benchmark + 轻量历史建模基座**，不是直接替代 APEX / AirNav 的方法主骨架，但很适合作为龙虾后续实验基础设施参考。

## 可借鉴的技术点
1. **多引擎统一接口**：把 UE、GTA、Google Earth、3DGS 统一到同一控制/感知接口，很适合作为空中 VLN 数据工厂模板。
2. **自动轨迹 + 自动指令生成**：利用点云、语义分割、A* 轨迹搜索和 VLM/LLM 指令生成，显著降低人工标注成本。
3. **关键帧感知历史压缩**：用动作变化点和 landmark grounding 选关键帧，再做 visual token merging，适合给主人当前机载 VLM 前端减负。
4. **real-to-sim 补强**：3DGS 场景和真实外场实验一起上，说明它对 sim-to-real 闭环是有意识设计的。

## 局限 / 不足
1. 方法贡献更多在平台、数据和轻量压缩，而不是提出全新的长程 explorer 主线。
2. OpenFly-Agent 仍基于 OpenVLA 式 action prediction，离“开放式语义搜索 + 复杂重规划”还有距离。
3. benchmark 以 4 DoF 离散动作为主，对真正 6-DoF 高动态无人机搜索仍偏保守。

## 对主人的直接启发
对龙虾最有价值的不是直接复现 OpenFly-Agent，而是吸收它的 **多场景数据生产链路** 和 **关键帧压缩策略**。短期建议把 OpenFly 当成 D06 的基础设施参考，与 **AirNav 的真实基准**、**APEX 的解耦 explorer** 拼成更完整的实验底座。

## 后续动作
