## 一、核心定位

当前系统不应只停留在「已有地图自动导航采集」，而应进一步升级为：

**语义目标驱动的闭环场景生成与数据扩展系统。**

把场景扩展从「导入更多地图」升级为：

```
任务需求 → 场景规格 → 程序化地图生成 → 仿真验证 → 自动采集 → 覆盖分析 → 场景再生成
```

重点不在天气、光照等表层随机化，而在：

- 地图结构如何规模化生成
- 建筑、道路、屋顶、巷道、障碍物如何程序化扩展
- LLM 如何参与语义规划
- 三维重建如何反哺仿真场景
- 失败样本如何驱动下一轮场景生成

---

## 二、总体架构：以 SceneSpec 作为统一中间层

不要让 LLM 或生成模型直接生成 UE 场景，而是先生成一个统一的结构化场景描述：

### SceneSpec：场景规格说明书

SceneSpec 负责描述：

- 场景边界
- 道路 / 巷道 / 开阔区 / 屋顶区
- 建筑体块
- 障碍物分布
- 语义目标
- 任务点
- 可飞行约束
- 采集路径约束
- 评测指标

### 整体架构

```
语义任务输入
  ↓
SceneSpec 场景规格
  ↓
地图 / 拓扑 / 语义编译器
  ↓
程序化场景生成后端
  ↓
UE / 仿真平台场景实例化
  ↓
可飞行性 / 可见性 / 碰撞 / 任务验证
  ↓
自动导航与数据采集
  ↓
覆盖率分析与失败挖掘
  ↓
下一轮 SceneSpec 修复与再生成
```

**SceneSpec 是系统的核心资产。** 以后无论接 UE PCG、Houdini、OSM、三维重建、3DGS、生成式 3D 模型，都统一转成 SceneSpec，再进入后续流水线。

## 三、推荐的三条场景扩展路线

### 路线 A：边界地图生成（最快落地）

**近期最推荐的主线。**

**输入：**
- 矩形 / 多边形边界
- 场景类型
- 障碍密度
- 建筑分布
- 任务需求
- 飞行约束

**输出：**
- 道路
- 巷道
- 建筑体块
- 屋顶平台
- 立面目标
- 障碍物
- 航线
- 采集点
- 语义标注

**适合生成：**
- 校园区块
- 园区道路
- 工业场站
- 楼宇夹缝
- 屋顶巡检区域
- 开阔区到狭窄通道的过渡场景
- 多高度 / 多遮挡 / 多转角跟踪场景

**工程最稳**，可控、可复现、方便批量生成，也最容易和自动导航采集结合。

---

### 路线 B：LLM 语义目标驱动场景

适合形成系统创新点。用户不直接指定地图，而是输入任务需求：

> "生成一批适合无人机跟随行人穿过狭窄巷道的场景，要求包含遮挡、转角、目标短暂丢失、重新出现和多高度视角。"

LLM 不直接摆放模型，而是生成结构化 SceneSpec：

```yaml
场景类型：城市巷道
区域结构：开阔区 → 窄巷 → 转角 → 屋顶边缘
目标：行人
遮挡物：墙体、车辆、立柱、树木
任务：目标跟随、重新识别、绕飞避障
约束：最小通行宽度、目标可见率、路径曲率、采集距离分布
```

**LLM 的真正价值不是「画地图」，而是：**
- 把任务需求翻译成场景约束
- 生成多样化场景组合
- 根据失败报告修复 SceneSpec
- 定向生成长尾难例

---

### 路线 C：真实重建回流与局部数字孪生

适合做 sim2real 和真实失败闭环。

```
真实场景图片 / 视频 / 无人机采集
  ↓
三维重建：Mesh / 3DGS / 点云
  ↓
提取真实结构：道路、墙面、建筑轮廓、屋顶、障碍物
  ↓
转成 SceneSpec
  ↓
程序化重建可控仿真场景
  ↓
在失败区域附近生成反事实变体
```

**Real2Spec，而不是简单 Real2UE。**

3DGS 更适合作为视觉背景或真实外观参考；真正承担碰撞、语义、任务逻辑和可飞行验证的，仍应是程序化生成的结构化场景。

## 四、建议的系统模块（9个）

1. **SceneSpec Schema** — 定义场景规格、任务规格、约束规格、资产标签、评测指标
2. **Map Skeleton Generator** — 生成地图骨架：边界、道路、巷道、建筑体块、屋顶、开阔区
3. **Semantic Layout Generator** — 生成语义布局：目标物、障碍物、跟踪对象、交互点、采集点
4. **Asset Instantiator** — 从资产库或生成式模型中选择建筑、障碍、目标物，并完成摆放
5. **UE Scene Builder** — 将 SceneSpec 编译为 UE 关卡、Actor、碰撞体、语义标签和采集接口
6. **Validator** — 检查场景是否可飞、是否穿模、路径是否连通、目标是否可见、任务是否可执行
7. **Auto Collection Planner** — 自动生成导航路径、跟踪路径、相机视角、采集 episode
8. **Dataset Analyzer** — 分析数据覆盖率：距离、高度、视角、遮挡、目标尺度、场景类型、失败区域
9. **Feedback Scene Planner** — 根据数据缺口和失败日志，生成下一批 SceneSpec

## 五、闭环机制：从「采数据」升级为「主动造数据」

最关键的是形成数据闭环：

```
自动生成场景
  ↓
自动导航采集
  ↓
统计数据分布
  ↓
发现覆盖不足
  ↓
生成定向难例
  ↓
重新采集
```

例如模型在以下情况表现差：

- 高俯视角
- 小目标 / 半遮挡 / 远距离跟踪
- 狭窄巷道
- 转角后目标重新出现
- 屋顶边缘目标
- 立面附近贴近飞行

系统就自动生成下一批场景：

- 增加屋顶边缘
- 增加栏杆 / 墙角遮挡
- 增加远距离目标
- 增加窄巷转角
- 增加开阔区到复杂区的过渡路径
- 增加目标短暂消失再出现的轨迹

**不是随机生成更多场景，而是根据模型失败主动生成有价值场景。**

## 六、关键评测指标

### 场景结构指标
- 道路 / 巷道连通率
- 可飞区域比例
- 最小通行宽度
- 障碍物密度
- 建筑体块多样性
- 屋顶 / 立面 / 开阔区覆盖率

### 仿真可用性指标
- 导入成功率
- 碰撞体有效率
- 穿模率
- NavGraph 连通率
- 采集路径可执行率
- 目标可见率

### 数据覆盖指标
- 距离分布
- 高度分布
- 视角分布
- 目标尺度分布
- 遮挡比例
- 路径曲率
- 速度分布
- 场景类型覆盖率

### 任务指标
- 导航成功率
- 目标跟踪成功率
- 目标重识别成功率
- 避障成功率
- 任务完成率
- 失败类型分布

## 七、MVP 落地路线

### Phase 0：边界地图生成器

**目标：**
- 输入边界和场景类型
- 自动生成低空场景
- 自动导入 UE
- 自动验证可飞性
- 自动生成采集路径

**优先实现：**
- 建筑体块生成
- 道路 / 巷道生成
- 屋顶区生成
- 障碍物采样
- 语义目标点生成
- 飞行路径采样
- 碰撞和可见性验证

### Phase 1：LLM SceneSpec Agent

自然语言任务 → SceneSpec → 程序化场景

**典型能力：**
- 根据任务生成场景需求
- 根据失败报告修改场景约束
- 根据数据缺口生成下一批难例

### Phase 2：真实重建回流

真实场景 → 重建 / 点云 / 3DGS → SceneSpec → 仿真变体

重点不是复刻真实场景，而是把真实场景中的结构和失败点转成可控的仿真生成规则。

### Phase 3：平台化 SceneOps

- 场景版本化
- 资产版本化
- 任务版本化
- 评测结果版本化
- 失败样本版本化

最终形成一个可持续扩展的数据工厂。

## 八、最精简的黄金流水线

```
Task Text
  ↓
SceneSpec
  ↓
Map / Topology Compiler
  ↓
Semantic Layout Generator
  ↓
Asset Instantiation
  ↓
UE Scene Builder
  ↓
Flyability / Visibility / Collision Validation
  ↓
Auto Navigation Collection
  ↓
Dataset Coverage Analysis
  ↓
Failure Mining
  ↓
SceneSpec Patch
  ↓
Regeneration
```

用 SceneSpec 把 LLM、程序化地图、真实重建、资产生成和自动采集统一起来，让场景扩展从「被动导入地图」变成「主动生成任务需要的世界」。

## 九、直接下发的实现清单（10条）

1. 定义 SceneSpec JSON Schema
2. 实现边界地图到拓扑图的 map2topo
3. 实现拓扑图到语义场景图的 topo2scenegraph
4. 实现建筑体块、道路、巷道、屋顶、障碍物的程序化生成
5. 接 UE 场景构建器，自动生成关卡和 Actor
6. 实现碰撞、可飞行性、目标可见性验证
7. 实现自动采集路径生成
8. 实现数据覆盖率统计
9. 实现失败样本分类
10. 实现失败驱动的 SceneSpec 再生成

## 十、最终判断

这套系统的主线应当是：

```
SceneSpec 中间表示
+ 程序化地图生成
+ LLM 语义规划
+ 真实重建回流
+ 生成式资产补全
+ 自动采集闭环
```

不要把重心放在「一句话生成完整 UE 世界」这种炫技路线。更稳、更能发文章、更能支撑 benchmark 的路线是：

**先用规则和程序化生成保证可控性，再用 LLM 提供语义多样性，再用真实重建和失败闭环提供 sim2real 价值。**
