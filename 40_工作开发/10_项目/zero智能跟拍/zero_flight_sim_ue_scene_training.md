## 1. 项目一句话定位

`zero_flight_sim` 是一个基于 Unreal Engine 5.7 与 AirSim 的飞行仿真平台，用于低空无人机仿真、人物跟随、传感器采集、场景交互和外部算法联调。

当前主线特点：

- 主 Unreal 工程：`Unreal/airworld`
- 引擎版本：Unreal Engine 5.7
- 对外通信方式：RPC
- 主要车辆类型：多旋翼无人机
- 坐标系主线：运行时 RPC API 使用 FLU
- 场景对象：无人机、人物、环境物体、调试标记、传感器数据
- 外部调用入口：PythonClient，后续可继续接 Matlab / ROS / 其他控制器

## 2. 系统全局架构

### 2.1 从使用者视角看系统

| 层级      | 主要内容                                      | 说明                    |
| ------- | ----------------------------------------- | --------------------- |
| 外部控制层   | PythonClient、Matlab、后续算法系统                | 发命令、读状态、采集数据          |
| RPC 接口层 | AirSim RPC Server / Client                | 外部系统与 UE 的通信边界        |
| 仿真编排层   | GameMode、SimMode、WorldSimApi              | 启动世界、注册车辆、管理对象、分发 API |
| 车辆与传感器层 | FlyingPawn、PawnSimApi、PIPCamera、Lidar、IMU | 无人机、相机、雷达、状态同步        |
| 场景内容层   | 地图、蓝图、人物、材质、模型                            | UE 里的可见世界和实验对象        |
| 项目扩展层   | zero_sim_up 人物系统、模型切换、行为组件                | 本仓库基于 AirSim 扩展出的能力   |

### 2.2 从运行链路看系统

1. UE 打开 `zero_sim_up.uproject`。
2. 当前关卡使用 `BP_AirsimGameMode`。
3. GameMode 初始化 AirSim 运行环境。
4. SimMode 创建或接管无人机 Pawn、相机、API Provider、录制等运行对象。
5. RPC Server 启动并监听外部连接。
6. PythonClient 连接 RPC Server。
7. 外部脚本控制无人机、查询人物、获取图像和传感器数据。
8. UE 场景持续渲染、仿真、采集和响应命令。

### 2.3 当前系统边界

- UE 负责真实场景、渲染、Actor 生命周期、物理移动和传感器生成。
- AirSim 插件负责仿真运行时、车辆接口、RPC 服务、相机传感器和世界 API。
- `zero_sim_up` 项目模块负责本项目特有的人物行为、人物模型切换和项目级扩展。
- PythonClient 负责任务编排、外部控制、数据采集和验证脚本。
- 大地图、导入资产、缓存产物不应该混入主线提交。

## 4. Unreal 工程启动与基础环境

### 4.1 开发环境

推荐环境：

- Unreal Engine 5.7
- Visual Studio 2022
- Python 3.10 或 3.11
- Git

Visual Studio 需要具备：

- C++ 游戏开发组件。
- MSVC v143 工具集。
- Windows SDK。
- CMake 工具链。

### 4.2 工程入口

| 入口 | 用途 |
|---|---|
| `Unreal/airworld/zero_sim_up.uproject` | UE 编辑器入口 |
| `Unreal/airworld/zero_sim_up.sln` | Visual Studio 编译入口 |
| `Unreal/airworld/Plugins/AirSim/Source/AirLib/AirLib.sln` | AirLib 独立编译入口 |

### 4.3 推荐启动顺序

1. 首次环境准备时，先编译 AirLib。
2. 再编译 `zero_sim_upEditor`。
3. 打开 `zero_sim_up.uproject`。
4. 打开目标关卡。
5. 确认 GameMode 为 `BP_AirsimGameMode`。
6. 点击 Play。
7. 使用 PythonClient 连接并验证。

## 5. UE 场景开发总流程

### 5.1 场景开发的目标

一个可用于仿真和外部 API 联调的场景，至少包含：

- 可运行的关卡。
- 正确的 AirSim GameMode。
- 一个或多个无人机 Pawn。
- 可选的人物目标。
- 可选的环境物体、障碍物、灯光和天气。
- 可识别的对象命名。
- 可被传感器采集的视觉环境。

### 5.2 标准场景搭建步骤

1. 创建或打开关卡。
2. 设置世界基础环境。
3. 配置光照、天空、地面、碰撞体。
4. 设置 GameMode。
5. 放置无人机蓝图。
6. 放置人物或目标对象。
7. 设置对象名称和标签。
8. 配置相机、传感器和必要的调试对象。
9. 保存关卡。
10. Play 后用 PythonClient 验证。

### 5.3 GameMode 是第一检查项

在目标关卡中需要检查：

- `World Settings`
- `GameMode Override`
- 选择 `BP_AirsimGameMode`

原因：

- AirSim 运行时初始化依赖 GameMode。
- RPC Server 的启动链路依赖当前 AirSim 运行时。
- PythonClient 只有在 UE Play 后才能连接到 RPC 服务。

### 5.4 地图与资产导入原则

场景资产可以来自：

- UE 自建几何。
- Datasmith 导入。
- Blender / FBX / glTF 等模型源。
- 本地大地图资产包。
- 项目已有 Content 目录资源。

建议原则：

- 先做轻量可运行关卡，再引入复杂地图。
- 大型导入地图应先作为本地资源验证，不直接纳入主线提交。
- BuiltData、缓存和 DerivedData 不作为源码资产管理重点。
- 场景中的重要对象需要规范命名，方便 RPC 查询。
- 演示场景不要依赖本机绝对路径。

### 5.5 场景命名建议

| 类型    | 命名建议                        | 目的           |
| ----- | --------------------------- | ------------ |
| 人物    | `Person1`、`Person_Target_A` | 方便外部脚本按名查询   |
| 无人机   | `Drone1`、默认空名车辆             | 方便多机扩展       |
| 调试绘制器 | `Painter1`                  | 供绘图 API 定位   |
| 目标物体  | `Target_*`                  | 方便检测、分割、任务生成 |
| 障碍物   | `Obstacle_*`                | 方便脚本筛选和调试    |

## 6. 无人机系统讲解

### 6.1 无人机在 UE 中是什么

在 UE 场景里，无人机主要表现为：

- 一个可放置的飞行器蓝图。
- 一个 Pawn。
- 一个与 AirSim 车辆 API 绑定的仿真对象。
- 一个挂载相机和传感器的移动平台。

常用飞行器蓝图位于：

- `Content/AirSim/BP_FlyingPawn.uasset`
- `Content/AirSim/BP_FlyingPawn_Hover.uasset`
- `Content/AirSim/BP_FlyingPawn_Fang.uasset`
- `Content/AirSim/BP_FlyingPawn_Gripper.uasset`

### 6.2 无人机运行链路

1. 场景中存在飞行器 Pawn。
2. SimMode 创建车辆 API 和 PawnSimApi。
3. Python 通过 RPC 发起飞控命令。
4. 多旋翼 API 接收期望速度、位置或姿态。
5. PawnSimApi 把命令转给 UE 侧运动链路。
6. UE 场景更新飞行器位置和相机位姿。
7. 传感器从当前位姿采集数据。

### 6.3 当前控制主线

当前主线是“外部 API 直驱 + 轻量运动学积分 + 响应整形”。

演讲时可以这样理解：

- 不以完整旋翼力学还原为第一目标。
- 优先保证 RPC 控制可用、响应稳定、数据采集链路顺畅。
- 外部控制器通过速度、位置、高度和 yaw 等 API 驱动无人机。
- UE 侧负责表现、碰撞、相机和传感器同步。

### 6.4 坐标系说明

本仓库运行时 RPC API 使用 FLU：

| 轴 | 含义 |
|---|---|
| X | 前 |
| Y | 左 |
| Z | 上 |

关键说明：

- 位置单位是米。
- 线速度单位是米每秒。
- 正 yaw 表示向左旋转。
- `vz > 0` 表示上升。
- 旧 AirSim 的 NED 转换工具仅保留给 legacy 脚本或地理转换使用。

### 6.5 场景里放置无人机时要注意

- 飞行器应位于可见、无遮挡、无穿模的位置。
- 起始位置附近应有足够空间。
- 如果有多机，命名和车辆配置需要清晰。
- 传感器朝向要与任务目标匹配。
- 场景碰撞体不应让无人机一开始就处于碰撞状态。

## 7. 人物系统讲解

### 7.1 人物系统定位

人物系统用于：

- 人机跟随实验。
- 目标跟踪。
- 数据采集中加入动态人体目标。
- 通过人物模型切换增强数据多样性。

人物系统不是完整游戏 AI 框架，也不应塞进无人机车辆仿真体系。

### 7.2 三层职责

| 层级 | 职责 |
|---|---|
| UE 人物蓝图与 C++ 角色 | 执行动作、移动、姿态、模型显示 |
| AirSim / RPC 桥接 | 查询人物状态、转发控制台命令 |
| 外部 Python 控制器 | 编排实验、控制无人机、触发人物行为 |

### 7.3 现有人物能力

当前人物系统支持：

- 站立。
- 行走。
- 奔跑。
- 蹲伏。
- 坐姿。
- 跳跃。
- 自动行为。
- 运行时模型切换。
- 按名字查询人物位置。
- 查询骨骼关键点和人物动作状态。

### 7.4 人物蓝图与资源

| 资源 | 位置 | 用途 |
|---|---|---|
| 人物蓝图 | `Content/Person/Blueprints/BP_ThirdPersonCharacter.uasset` | 放入场景的人物 Actor |
| 模型配置 | `Content/Person/Blueprints/DA_HumanModels.uasset` | 配置可切换外观模型 |
| 输入配置 | `Content/Person/Input` | 人物输入动作 |
| 人物 UI | `Content/Person/UI` | 人物相关界面资产 |

### 7.5 自动行为系统

自动行为组件位于项目 C++ 模块中：

- `ZeroHumanAutoBehaviorComponent`

它的设计目标：

- 让人物在一定范围内自主活动。
- 支持站立、行走、奔跑、蹲伏、坐姿等行为段。
- 可通过外部命令启动、停止、切换或查询状态。
- 行为参数可以在蓝图侧调整。

演讲时可以用一句话概括：

> 自动行为系统让人物成为“可控又可随机化的动态目标”，服务于无人机跟随和数据采集。

### 7.6 模型切换系统

模型切换系统用于在运行时改变人物外观。

核心思想：

- 驱动体负责动画逻辑。
- 显示体负责外观表现。
- 同骨架模型可以直接跟随。
- 异骨架模型可以通过 Retarget 动画蓝图跟随。

适合演示的场景：

- 同一个运动轨迹中随机换人物外观。
- 每个 episode 开始前随机选择人物模型。
- 用不同体型、衣着、骨架提高视觉数据多样性。

### 7.7 人物命名约定

如果要运行人物跟随示例，场景中的人物建议命名为：

- `Person1`

原因：

- Python 示例会按名称查询人物位置。
- 名称不匹配时，跟随脚本无法找到目标。

## 8. 相机与传感器系统讲解

### 8.1 传感器的角色

传感器系统负责把 UE 世界转换成外部算法可用的数据。

常见数据包括：

- RGB 图像。
- 深度图。
- 语义分割。
- 表面法线。
- 红外。
- 光流。
- 检测框。
- 激光雷达点云。
- IMU。
- GPS / 地理信息。
- 距离传感器。

### 8.2 相机链路

相机链路可以概括为：

1. 无人机 Pawn 挂载相机。
2. PIPCamera 管理具体相机和渲染目标。
3. RenderRequest 负责图像读取。
4. UnrealImageCapture 聚合图像请求。
5. WorldSimApi 通过 RPC 暴露给外部。
6. PythonClient 解码图像并用于显示或保存。

### 8.3 支持的视觉类型

| 类型 | 用途 |
|---|---|
| Scene | 常规 RGB 图像 |
| DepthPlanar | 平面深度 |
| DepthPerspective | 透视深度 |
| DepthVis | 可视化深度 |
| DisparityNormalized | 归一化视差 |
| Segmentation | 语义分割 |
| SurfaceNormals | 表面法线 |
| Infrared | 红外 |
| OpticalFlow | 光流 |
| OpticalFlowVis | 可视化光流 |

### 8.4 全景相机能力

当前 AirSim 插件中已经包含全景捕获组件：

- 可按相机名称识别全景请求。
- 可设置全景分辨率。
- 可触发全景采集。
- 可通过 RPC 拉取全景图像。

适合的使用场景：

- 环境观察。
- 场景质量检查。
- 数据采集中的全局上下文图。
- 演示无人机当前位置周围的环境。

### 8.5 检测与分割

检测能力主要用于：

- 调试目标是否可见。
- 获取规则化 2D / 3D 检测信息。
- 生成或验证标注数据。
- 支撑目标跟踪和场景理解实验。

分割能力主要用于：

- 给对象设置分割 ID。
- 按对象类别或名称生成语义图。
- 验证场景对象是否被正确识别。

### 8.6 传感器诊断

PythonClient 中有传感器诊断脚本，用于：

- 一次性测试多种图像类型。
- 测试物理传感器数据。
- 测试 LiDAR。
- 测试检测接口。
- 生成报告和可视化图。

会议演示建议：

- 不讲脚本内部实现。
- 讲它的用途：确认场景和传感器是否完整可用。
- 讲输出结果如何帮助定位相机、配置、命名或场景问题。

## 9. 外部 API 使用方式

### 9.1 外部 API 总原则

当前对外 API 统一走 RPC。

讲解时需要强调：

- UE 必须先进入 Play 状态。
- 当前地图必须正确使用 AirSim GameMode。
- PythonClient 连接的是本机 RPC Server。
- 端口默认是 `41452`。
- 不建议每个脚本各自硬编码连接配置。

### 9.2 PythonClient 连接入口

推荐使用：

- `PythonClient/airsim/connection.py`

默认参数：

| 参数      | 默认值         |
| ------- | ----------- |
| IP      | `127.0.0.1` |
| Port    | `41452`     |
| Timeout | `3600`      |

可通过环境变量覆盖：

- `ZERO_SIM_RPC_IP`
- `ZERO_SIM_RPC_PORT`
- `ZERO_SIM_RPC_TIMEOUT`

### 9.3 API 能力分组

| API 类型 | 能力举例                 | 面向场景开发的意义  |
| ------ | -------------------- | ---------- |
| 连接与控制  | 连接、版本、暂停、继续、重置       | 判断仿真是否在线   |
| 无人机控制  | 起飞、降落、速度、位置、高度、姿态    | 执行飞行任务     |
| 状态读取   | 位姿、速度、碰撞、环境、车辆列表     | 记录轨迹和调试    |
| 图像传感器  | RGB、深度、分割、光流、全景      | 采集视觉数据     |
| 物理传感器  | IMU、LiDAR、距离、GPS     | 采集多模态数据    |
| 世界对象   | 查询对象、生成对象、销毁对象、姿态、缩放 | 动态构建和修改场景  |
| 材质与环境  | 材质替换、光照、天气、时间        | 做域随机化和场景变化 |
| 检测与标注  | 检测过滤、检测框、分割 ID       | 支撑标注和评估    |
| 人物桥接   | 查询人物、骨骼点、动作状态、控制台命令  | 支撑人机交互实验   |
| 调试绘制   | 点、线、箭头、坐标轴、文字        | 让任务轨迹可视化   |

## 10. 场景开发实战流程

### 10.1 第一类：最小可运行飞行场景

目标：

- 场景能启动。
- 无人机能起飞。
- Python 能连接。
- 图像能获取。

步骤：

1. 新建或复制一个轻量关卡。
2. 设置 `BP_AirsimGameMode`。
3. 放置 `BP_FlyingPawn` 或 `BP_FlyingPawn_Hover`。
4. 放置基础地面、灯光和摄像环境。
5. 保存关卡。
6. Play。
7. 用基础连接脚本验证。
8. 用图像脚本验证相机。

讲解重点：

- 不要第一步就上复杂大地图。
- 先把运行链路跑通。
- 先解决 GameMode、Pawn、RPC、相机四件事。

### 10.2 第二类：人物跟随场景

目标：

- 场景中有可查询人物。
- 无人机能读取人物位置。
- 外部脚本能控制无人机跟随。

步骤：

1. 在最小飞行场景基础上放入 `BP_ThirdPersonCharacter`。
2. 将人物命名为 `Person1`。
3. 确认人物没有被遮挡或放在不可达位置。
4. 可选：启用自动行为。
5. Play。
6. 运行人物跟随脚本。
7. 用调试绘图或日志观察跟随效果。

讲解重点：

- 人物系统和无人机系统是解耦的。
- 人物是外部控制器的目标，不是无人机仿真体系的一部分。
- 命名规范决定外部脚本能否找到对象。

### 10.3 第三类：传感器采集场景

目标：

- 场景能稳定输出多模态传感器。
- 图像、深度、分割、LiDAR 等数据可用于实验。

步骤：

1. 场景中放置有视觉纹理和几何结构的环境。
2. 确认光照、材质和碰撞合理。
3. 配置相机默认参数。
4. 确认需要检测或分割的对象命名清晰。
5. Play。
6. 运行传感器诊断。
7. 根据报告调整场景、材质或相机。

讲解重点：

- 传感器质量依赖场景资产质量。
- 深度和分割需要几何、材质、对象 ID 等配合。
- 传感器诊断是场景交付前的质量门槛。

### 10.4 第四类：动态对象和任务场景

目标：

- 通过外部 API 动态修改场景。
- 支持任务生成、对象生成、材质替换、天气变化。

可以演示的能力：

- 查询场景对象。
- 按名称获取对象位置。
- 动态生成对象。
- 动态销毁对象。
- 修改对象姿态或缩放。
- 替换材质或纹理。
- 调整天气和时间。
- 用绘图 API 显示轨迹或目标点。

讲解重点：

- UE 场景不只是静态地图。
- RPC 让外部实验系统可以动态编排场景。
- 动态能力适合做 episode、数据增强和任务随机化。

- 不要把人物 AI 硬塞进无人机车辆体系。

- 对外能力优先通过 RPC 暴露。
- 场景开发先保证最小闭环，再增加复杂资产和任务逻辑。

## 12. 常见问题和排查路径

### 12.1 Python 无法连接 UE

优先检查：

- UE 是否已经点击 Play。
- 当前关卡是否使用 `BP_AirsimGameMode`。
- RPC 端口是否为 `41452`。
- 防火墙是否拦截。
- Python 是否使用统一连接入口。

### 12.2 无人机不响应控制

优先检查：

- 是否启用了 API 控制。
- 是否已解锁或起飞。
- 场景中是否有可控飞行器 Pawn。
- 命令使用的 vehicle name 是否正确。
- 是否处于碰撞、卡住或被其他逻辑覆盖状态。

### 12.3 图像为空或传感器异常

优先检查：

- 相机名称是否正确。
- 图像类型是否被支持。
- 当前场景是否在 Play 状态。
- 相机是否被遮挡或未正确挂载。
- settings 中相机参数是否合理。
- 运行传感器诊断脚本确认整体状态。

### 12.4 人物跟随不工作

优先检查：

- 人物是否放入当前关卡。
- 人物是否命名为 `Person1`。
- 人物是否被隐藏或未生成。
- 外部脚本是否能通过对象查询 API 找到它。
- 自动行为是否让人物离开预期区域。

### 12.5 场景提交变得很大

优先检查：

- 是否误提交了大地图包。
- 是否误提交了 BuiltData。
- 是否误提交了 DerivedDataCache 或 Saved。
- 是否误提交了本地导入资产。
- `.gitignore` 是否已覆盖本地大资产路径。

## 14. 场景交付检查清单

### 14.1 基础可运行

- 当前关卡能打开。
- 当前关卡能 Play。
- GameMode 已设置为 AirSim GameMode。
- 场景中至少有一个可控无人机。
- PythonClient 能连接。
- 基础飞行命令有效。

### 14.2 人物和对象

- 人物已放入场景。
- 关键人物有稳定名称。
- 目标对象命名清晰。
- 需要检测或分割的对象可被识别。
- 人物自动行为不会导致目标离开可观测区域。

### 14.3 传感器

- RGB 图像正常。
- 深度图正常。
- 分割图正常。
- LiDAR 或 IMU 数据正常。
- 相机视角符合任务需要。
- 全景或特殊相机如需使用，已验证。

### 14.4 数据采集

- 采集脚本能运行。
- 输出目录明确。
- 轨迹和图像能对应。
- 时间、位姿、传感器数据没有明显异常。
- 失败场景有日志或诊断结果。

### 14.5 版本管理

- 未提交 `Binaries`、`Intermediate`、`Saved`、`DerivedDataCache`。
- 未提交本地大地图包。
- 未提交临时采样结果。
- `.gitignore` 覆盖本地导入大资产。
- 提交说明能说明场景或功能改动目的。

## 16. 一页总结

`zero_flight_sim` 的核心是：

- UE 负责真实感场景和运行时对象。
- AirSim 插件负责仿真、车辆、相机、传感器和 RPC。
- `zero_sim_up` 模块负责项目特有人物系统和扩展能力。
- PythonClient 负责外部控制、采集和诊断。
- 场景开发先跑通最小闭环，再增加人物、传感器、动态对象和复杂地图。
- 对外能力统一走 RPC。
- 运行时坐标系按 FLU 理解。
- 大地图和本地导入资产要谨慎管理，避免污染主线提交。

会议收束语：

> 这个项目不是单纯的 UE 场景，也不是单纯的 AirSim 示例，而是一套“UE 场景资产 + AirSim 仿真运行时 + 外部 API 控制 + 数据采集验证”的闭环平台。后续所有开发都应该围绕这个闭环维护清晰边界。
