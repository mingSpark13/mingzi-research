# 空中操作数据集设计草案 v0.1

## 设计目标
为当前空中操作/空中机械臂数据集工作建立一套可扩展、可训练、可评测、可部署回灌的统一数据结构。目标不是只收演示视频，而是沉淀成支持 VLA / planner / evaluator / safety gate 的基础设施。

## 设计原则
1. **任务优先**：先定义任务 taxonomy，再决定采集内容
2. **具身显式化**：无人机基座与机械臂约束必须进入数据字段
3. **失败样本有价值**：不只保留成功演示
4. **关键帧友好**：为时序压缩和轻量训练预留索引
5. **可自动扩展**：后续能接自动轨迹生成、自动指令生成、自动标注

## 一、任务 taxonomy（第一版）

### T1 基础接近与对位
- 目标搜索后接近物体
- 对齐抓取位姿
- 靠近插槽/把手/开关等交互位点

### T2 基础抓取与释放
- 抓取静态物体
- 搬运后释放
- 指定区域放置

### T3 接触式精细操作
- 插入
- 挂接
- 按压
- 旋拧/拨动

### T4 长时程组合任务
- 搜索 → 接近 → 抓取 → 搬运 → 对位 → 安装
- 搜索 → 识别 → 语义决策 → 执行

### T5 干扰与恢复任务
- 风扰动下保持操作
- 抓取失败后恢复
- 目标丢失后重定位
- 接触失败后重规划

## 二、每条样本的最小单元
建议最小单元不是单帧，而是 **episode / segment 双层结构**。

### Episode
完整任务过程，如：
“发现目标杯子并抓取，再放到指定托盘”

### Segment
按阶段切分：
1. 搜索
2. 接近
3. 对位
4. 接触/抓取
5. 搬运
6. 放置/恢复

这样后续更适合：
- 训练分阶段策略
- 做失败定位
- 做关键帧抽取
- 做长时程 credit assignment

## 三、建议数据字段

### 1. 元信息
- episode_id
- segment_id
- scene_id
- task_id
- task_type
- environment_type
- operator_id
- collection_mode（teleop / scripted / mixed）
- success
- failure_type

### 2. 语言信息
- instruction_text
- subgoal_text
- object_name
- object_attributes
- target_location
- task_stage_label

### 3. 视觉信息
- rgb_front
- rgb_wrist / end-effector view（如有）
- depth
- segmentation / mask（可选）
- pointcloud（可选）
- keyframe_index
- key_event_label

### 4. 平台状态
- uav_position
- uav_orientation
- uav_linear_velocity
- uav_angular_velocity
- arm_joint_positions
- arm_joint_velocities
- end_effector_pose
- gripper_state
- battery / payload（可选）

### 5. 控制与动作
- high_level_action
- low_level_command
- action_chunk
- controller_mode
- replanning_flag
- intervention_flag

### 6. 目标与环境状态
- object_pose
- object_state
- contact_state
- obstacle_state
- wind_state
- lighting/weather

### 7. 可执行性 / 安全字段
- tracking_error
- stability_margin
- control_cost
- constraint_violation
- safety_gate_trigger
- recoverability_score

## 六、采集策略建议

### 第一阶段：高质量小规模人工遥操作
目标：先把 schema 和任务定义跑通
- 聚焦 3 到 5 类核心任务
- 每类少量高质量 episode
- 明确记录失败类型

### 第二阶段：模板化扩展
目标：增加场景、目标物、扰动条件
- 固定任务骨架
- 扩展物体、位置、背景、天气、光照

### 第三阶段：自动化生成 + 人类修正
目标：引入 OpenFly 风格的数据工厂思路
- 自动轨迹生成
- 自动子目标生成
- 自动标注阶段标签
- 人类只做修正与质检

## 七、近期最值得立刻落地的 5 件事
1. 先定任务 taxonomy v1
2. 先定 episode / segment 双层数据结构
3. 明确 failure_type 枚举表
4. 在采集脚本里加入关键帧索引与 task_stage_label
5. 把无人机稳定性/约束相关字段单独列为必填

## 八、不要踩的坑
- 只存成功样本
- 只存视频，不存状态和控制量
- 把空中平台当成普通机械臂
- 先猛采再想 schema
- 不记录失败恢复过程

## 九、后续扩展方向
- 接 AirVLA 式 benchmark 任务定义
- 接 UMI-on-Air 式可执行性反馈
- 接 OpenFly 式自动指令和关键帧压缩
- 为 evaluator / safety gate / planner 单独导出训练视图

## 十、当前一句话结论
**先用 AIR-VLA 定任务，用 UMI-on-Air 定约束与失败反馈，用 OpenFly 定自动化与数据工厂。**
