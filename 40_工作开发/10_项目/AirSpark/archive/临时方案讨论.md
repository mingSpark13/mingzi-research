你的判断我认为是**对的，而且应该成为 AirSpark P3 的方向修正**：

> **不要把 AirSpark 做成一个“大而全 UE 程序化场景生成器”；应该把重点转到“面向低空任务的语义地图、区域层级、任务推理轨迹和语言标注数据工厂”。**

更尖锐一点说：  
**场景生成是供给侧，标注与任务构造才是 AirSpark 的护城河。**  
UE PCG、Fab 城市场景包、UnrealLLM/AutoUE 这类东西后面会越来越成熟，自己重造一套 mini-PCG 引擎确实性价比不高。AirSpark 更应该做的是：无论场景来自 School 地图、Fab、PCG、3DGS 重建，还是未来真实场景回流，都能把它自动加工成可训练 VLN/VLA/推理记忆模型的数据。

你上传的 v0.7 文档里其实已经给了这个转向的基础：AirSpark 已完成 P0/P1/P2，P3 当前目标是 SceneSpec、QA Dashboard、Failure Mining、Regeneration Planner；同时数据格式里已经预留了 `map_context / semantic_target`、`high_level_instruction / subtask_instruction / atomic_skill_label`，说明系统已经有“地图上下文 + 语言 + 任务标签”的接口，只是还没有把它拔高成主线。

# 1. 方向调整：从 SceneOps-heavy 改成 AnnotationOps / MapOps

我建议 P3 不再叫“程序化场景生成阶段”，而改成：

> **P3：AnnotationOps — 低空语义地图与任务标注数据工厂**

SceneSpec 还保留，但它的角色降级：

```text
原来：
SceneSpec = 驱动 UE 程序化造世界的主抽象

建议：
SceneSpec = 记录场景来源、资产版本、区域层级、任务配置、标注策略的统一元数据
```

也就是说，SceneSpec 不负责“造场景”，而负责“描述和治理场景”。

新的核心对象应该是：

```text
SceneSpec
  ├── scene_source：School / Fab / PCG / 3DGS / RealScan
  ├── region_graph：楼、楼层、教室、走廊、操场、篮球场等区域层级
  ├── object_graph：门、窗、桌椅、篮球架、垃圾桶等对象实例
  ├── relation_graph：包含、相邻、连接、可见、支撑、位于左侧等关系
  ├── task_graph：搜索、接近、观察、绕飞、操作、恢复等任务过程
  ├── language_pack：自然语言指令、区域描述、目标描述、过程描述
  └── qa_report：标注置信度、冲突项、待人工确认项
```

这样一来，你后续接任何场景生成工具都可以：

```text
外部场景来源
    ↓
AirSpark Semantic / Region / Task AnnotationOps
    ↓
VLN/VLA/推理记忆训练数据
```

这比自己做 PCG 更有论文价值。

# 2. 你真正想做的不是“语言润色”，而是“可训练的语义记忆监督”

你说得很准，LLM 辅助标注不能只停留在：

```text
"Navigate to observe 这是一个教室的门"
    ↓
"飞到教室门附近观察"
```

这只是最浅层的 instruction augmentation。

真正有价值的是让每条 episode 带上：

```text
1. 这个物体属于哪个区域？
2. 这个区域属于哪个更大的区域？
3. 这个目标和其他目标有什么空间关系？
4. UAV 在探索过程中什么时候发现了什么？
5. 当前地图记忆中有哪些已知/未知区域？
6. 为了完成任务，中间应该推理出哪些子目标？
7. 轨迹每一步为什么朝这个方向飞？
8. 失败时是目标不可见、路径不可达、区域误判，还是观察点不合理？
```

也就是从：

```text
instruction + trajectory
```

升级成：

```text
instruction + trajectory + semantic map + region hierarchy + memory updates + subgoal reasoning labels
```

这才可能训练出你想要的 VLN 推理记忆能力。

MapNav 这类工作已经在往这个方向走：它用 **Annotated Semantic Map** 替代大量历史帧作为 VLN 的记忆表示，让 VLM 不只是看当前图像，而是看带文本标注的地图记忆。([arXiv](https://arxiv.org/abs/2502.13451?utm_source=chatgpt.com "MapNav: A Novel Memory Representation via Annotated Semantic Maps for VLM-based Vision-and-Language Navigation"))  
VLMaps 更早的思路是把视觉语言特征融合进 3D 地图，使地图可以被自然语言索引，例如定位 landmark 或空间关系目标。([GitHub](https://github.com/vlmaps/vlmaps?utm_source=chatgpt.com "vlmaps/vlmaps: [ICRA2023] Implementation of Visual ..."))  
所以你这个方向不是瞎想，是当前 VLN/VLA 很核心的一条线。

# 3. AirSpark 应该建立的核心数据结构：RegionGraph + ObjectGraph + TaskTrace

我建议你把语义系统拆成三张图。

---

## 3.1 RegionGraph：区域层级图

这是你提到的“教室、楼房、操场、篮球场嵌套”的核心。

它不是简单标签，而是一个层级结构：

```json
{
  "region_id": "region_playground_01",
  "name": "操场",
  "type": "playground",
  "parent": "campus_main",
  "children": ["region_basketball_court_01", "region_running_track_01"],
  "polygon_2d": [[...]],
  "height_range": [0.0, 20.0],
  "semantic_tags": ["outdoor", "sports_area", "open_area"],
  "navigable": true
}
```

篮球场是操场的子区域：

```json
{
  "region_id": "region_basketball_court_01",
  "name": "篮球场",
  "type": "basketball_court",
  "parent": "region_playground_01",
  "children": ["object_hoop_01", "object_hoop_02"],
  "semantic_tags": ["sports", "court", "flat_ground"],
  "navigable": true
}
```

教室门比较特殊，它不是简单“属于教室”，而是一个**边界对象 / portal object**：

```json
{
  "object_id": "SM_DoorClassroom2",
  "category": "door",
  "primary_region": "corridor_2f_left",
  "connected_regions": ["corridor_2f_left", "classroom_203"],
  "relation_type": "portal",
  "affordances": ["enter", "inspect", "open"]
}
```

这点非常重要。很多对象不是单区域归属，而是连接多个区域：

|对象|正确语义|
|---|---|
|门|连接走廊和教室|
|窗|属于建筑外立面，同时关联室内房间|
|楼梯|连接楼层|
|校门|连接校内和校外道路|
|篮筐|属于篮球场|
|跑道|属于操场|
|垃圾桶|位于走廊/操场/楼门口|
|标志牌|属于区域入口或路径节点|

所以你的区域标签不能做成扁平分类，必须做成**层级 + 关系**。

这可以对标 HOV-SG 这类 hierarchical open-vocabulary 3D scene graph：它把 3D 场景组织成 floor、room、object 等层级概念，并用开放词汇特征支持语言导航。([Robotics: Science and Systems](https://roboticsconference.org/2024/program/papers/77/?utm_source=chatgpt.com "Hierarchical Open-Vocabulary 3D Scene Graphs for ..."))

---

## 3.2 ObjectGraph：对象实例图

对象图记录每个可见、可操作、可导航、可作为 landmark 的对象：

```json
{
  "object_id": "SM_Window_01",
  "category": "window",
  "nouns": ["窗户", "玻璃窗"],
  "attributes": ["二楼", "白框", "长方形", "透明"],
  "affordances": ["inspect", "observe", "approach"],
  "region_id": "building_A_facade_east",
  "bbox_3d": {...},
  "observe_distance_m": 3.0,
  "visibility_anchors": [...]
}
```

这里 LLM/VLM 的作用不是凭空造标签，而是把已有的 UE 信息补全为可训练语义。

比如现在你有：

```json
{
  "semantic_name": "门",
  "target_object_id": "SM_DoorClassroom2"
}
```

LLM 可以补成：

```json
{
  "canonical_name": "教室门",
  "category": "door",
  "nouns": ["门", "教室门", "入口"],
  "attributes": ["室内", "走廊侧", "矩形", "可进入"],
  "affordances": ["接近观察", "进入教室", "检查门牌"],
  "possible_regions": ["走廊", "教室入口"],
  "language_aliases": [
    "走廊左侧的教室门",
    "二楼教室入口",
    "靠近墙边的那扇门"
  ]
}
```

这才是有训练价值的语义标注。

---

## 3.3 TaskTrace：轨迹过程推理图

这是训练“推理记忆能力”的关键。

不要只保存：

```text
instruction: 找到教室门
trajectory: xyz...
```

要保存每个阶段的结构化过程标签：

```json
{
  "task_id": "nav_observe_door_001",
  "instruction": "飞到二楼走廊左侧的教室门附近观察",
  "target": "SM_DoorClassroom2",
  "stages": [
    {
      "stage": "search_region",
      "goal": "进入二楼走廊区域",
      "expected_memory": ["building_main", "corridor_2f"],
      "success_condition": "corridor_2f visible or reached"
    },
    {
      "stage": "localize_target",
      "goal": "在走廊左侧寻找教室门",
      "expected_memory": ["door candidates", "left wall"],
      "success_condition": "target door visible"
    },
    {
      "stage": "approach_observe",
      "goal": "接近到推荐观察距离",
      "expected_memory": ["target bbox", "observe pose"],
      "success_condition": "distance within 2.5-3.5m and target in view"
    }
  ]
}
```

训练 VLN/VLA 时，这些字段可以变成：

```text
输入：当前图像 + 当前地图记忆 + 高层指令
监督：
  当前子目标是什么？
  应该更新哪些地图记忆？
  下一个 waypoint / action 是什么？
  当前失败风险是什么？
```

这比单纯模仿轨迹强得多。

# 4. 自动语义地图构建：推荐三条路径并行

你提出“让 UAV 探索完整地图后，结合 LLM、SAM3 或分割模型自行建立语义地图”，这个可以做，但要注意：**仿真内和真实世界的最优路线不一样**。

---

## 路径 A：仿真内优先使用 UE 真值，别浪费 SAM

在 UE 里，最可靠的真值来自：

```text
Actor metadata
SemanticComponent
Instance ID render pass
Depth
Camera pose
Collision mesh
Navigation/collision volume
```

这比 SAM / VLM 猜图像强得多。

所以仿真内第一阶段应该是：

```text
UE Actor / SemanticComponent
    ↓
ObjectGraph 真值
    ↓
RegionVolume / 手工粗标区域
    ↓
RegionGraph 真值
    ↓
自动生成语言、任务、轨迹、推理标签
```

这一步最稳，最适合 P3。

---

## 路径 B：真实/未知场景使用 SAM3 + 多视角融合

SAM3 现在的定位已经不是老 SAM 那种只切 mask，它是可以基于**文本概念或视觉示例**进行图像/视频中的目标检测、分割和跟踪的模型；Meta 官方 repo 也说 SAM3 支持 text prompt 或 visual prompt，并能在图像和视频中检测、分割、跟踪对象。([GitHub](https://github.com/facebookresearch/sam3?utm_source=chatgpt.com "facebookresearch/sam3: The repository provides code for ..."))  
SAM3.1 又重点提升了视频处理效率，适合长视频/探索轨迹。([Meta AI](https://ai.meta.com/blog/segment-anything-model-3/?utm_source=chatgpt.com "SAM 3.1: Faster and More Accessible Real-Time Video ..."))

但要注意：

> SAM3 给你的是 2D/视频 object mask，不会自动给你可靠的 3D 区域层级。

所以真实/未知场景的自动建图流程应该是：

```text
UAV 探索轨迹
    ↓
RGB / Depth / Pose / IMU
    ↓
SAM3 / Grounded-SAM / YOLO-seg 得到多帧 2D mask
    ↓
利用深度 + 相机位姿投影到 3D
    ↓
多视角关联，聚合成 3D object instance
    ↓
CLIP/SigLIP/DINO 特征聚合，得到 open-vocabulary object embedding
    ↓
LLM/VLM 根据 top-down map 和 object list 做语义命名与区域关系推断
```

这可以参考 ConceptGraphs 和 OpenMask3D 的思路。ConceptGraphs 就是利用 2D foundation model 输出，通过多视角关联构建 open-vocabulary 3D scene graph。([arXiv](https://arxiv.org/abs/2309.16650?utm_source=chatgpt.com "ConceptGraphs: Open-Vocabulary 3D Scene Graphs for ..."))  
OpenMask3D 则用 class-agnostic 3D mask，加上多视角 CLIP 特征聚合，实现开放词汇 3D 实例分割。([开放评审](https://openreview.net/forum?id=8vuDHCxrmy&utm_source=chatgpt.com "OpenMask3D: Open-Vocabulary 3D Instance Segmentation"))

你可以把它们的思想迁移到低空场景。

---

## 路径 C：区域不是靠 SAM 切出来，而是靠“几何 + 语义 + 拓扑”推断出来

这是最容易踩坑的地方。

SAM 能切出：

```text
门、窗、车、人、树、篮球架、道路标线
```

但它很难直接切出可靠的：

```text
教室
走廊
教学楼
操场
篮球场
施工区域
可飞行通道
巡检区域
```

区域通常不是单个视觉对象，而是空间集合。  
所以区域生成要靠：

```text
1. 几何边界
   墙、地面、道路边缘、围栏、建筑轮廓、区域 polygon

2. 拓扑关系
   门连接房间和走廊，路连接路口，楼梯连接楼层

3. 语义对象集合
   篮筐 + 球场线 + 平整地面 → 篮球场
   黑板 + 桌椅 + 门牌 → 教室
   多栋建筑 + 道路 + 操场 → 校园

4. LLM 归纳
   根据对象集合和空间布局，命名区域并建立层级
```

所以你的自动区域标注模块应该长这样：

```text
Object Instances
    ↓
Spatial Clustering
    ↓
Boundary / Connectivity Analysis
    ↓
Region Candidate Generation
    ↓
LLM Region Naming + Hierarchy Inference
    ↓
Human Review / Rule Validation
```

例如：

```text
检测到：
  basketball_hoop × 2
  court_line markings
  flat rectangular ground
  inside playground polygon

推断：
  region_basketball_court_01
  parent = region_playground_01
  type = basketball_court
```

又例如：

```text
检测到：
  door
  classroom sign
  desks/chairs through window or doorway
  connected to corridor

推断：
  classroom_203
  parent = building_A_floor_2
  connected_by = SM_DoorClassroom2
```

# 5. 低空语义地图应该不是普通 semantic map，而是五层 MapMemory

为了训练 VLN 推理记忆能力，我建议你不要只做 top-down semantic map。应该做一个分层记忆结构：

```text
AirSpark MapMemory
├── Metric Map
│   ├── occupancy grid / ESDF / height map
│   └── collision / no-fly zone
├── Object Map
│   ├── object instance
│   ├── 3D bbox / mask / visibility
│   └── affordance / observe pose
├── Region Map
│   ├── room / corridor / building / playground / court
│   ├── containment hierarchy
│   └── region connectivity
├── Relation Graph
│   ├── left_of / near / inside / connected_to / facing
│   └── visible_from / reachable_from / observe_from
└── Task Memory
    ├── discovered targets
    ├── candidate goals
    ├── visited / unvisited regions
    ├── subgoal status
    └── failure / uncertainty
```

这个东西才是给 VLN 模型看的“记忆”。

你后面可以训练两类模型：

### 1. Map-conditioned VLN / VLA

输入：

```text
当前 RGB
当前 state
高层 instruction
MapMemory summary
```

输出：

```text
next waypoint / velocity / action chunk
```

### 2. Memory updater / reasoning model

输入：

```text
当前观察
上一时刻 MapMemory
```

输出：

```text
新增对象
新增区域
目标候选更新
下一步探索区域
```

这就比单纯 imitation learning 更接近“会探索、会记忆、会推理”。

# 6. LLM 在这里具体做什么？我建议分成 7 个模块

---

## 模块 1：Semantic Patch Generator

目标：补全 UE 里已有 SemanticComponent 的空字段。

输入：

```json
{
  "actor_name": "SM_DoorClassroom2",
  "semantic_name": "门",
  "category": "door",
  "level_name": "School",
  "nearby_objects": ["SM_Wall_02", "SM_ClassroomSign_01"]
}
```

输出：

```json
{
  "canonical_name": "教室门",
  "nouns": ["门", "教室门", "入口"],
  "attributes": ["走廊侧", "室内入口", "矩形"],
  "affordances": ["接近观察", "进入教室", "检查门牌"],
  "observe_distance_m": 2.5,
  "task_roles": ["navigation_target", "inspection_target", "portal"]
}
```

这一步优先级最高，因为它直接提升现有 School 地图的数据价值。

---

## 模块 2：RegionGraph Builder

目标：建立区域层级。

短期可以手工给大区域画 volume/polygon：

```text
building_A
  ├── floor_1
  ├── floor_2
  │   ├── corridor_2f
  │   ├── classroom_201
  │   └── classroom_203
playground
  ├── basketball_court
  └── running_track
```

然后 LLM 负责补自然语言别名和区域描述：

```json
{
  "region_id": "corridor_2f",
  "aliases": ["二楼走廊", "教学楼二楼过道", "二层通道"],
  "description": "连接多个教室门的狭长室内通道，适合作为搜索教室门和楼梯的导航区域。",
  "typical_objects": ["door", "window", "sign", "wall"]
}
```

中期再让自动分割/探索地图生成 region candidate。

---

## 模块 3：Object-to-Region Resolver

目标：判断物体属于哪个区域、是否跨区域、是否是边界连接物。

规则优先，LLM 辅助：

```text
如果 object bbox center 在 region polygon 内 → belongs_to
如果 object 与两个 region 边界相交，category=door/window/gate → connects
如果 object 位于外立面 polygon → belongs_to facade region
如果 object 位于 court polygon 内 → belongs_to basketball court
```

LLM 处理模糊情况：

```text
“这扇门既在走廊边界，也通向教室，它应该如何标注？”
→ primary_region=corridor
→ connected_regions=[corridor, classroom]
→ role=portal
```

这个模块非常关键，因为它决定模型能否学会“目标在哪个区域里找”。

---

## 模块 4：Exploration-to-Map Annotator

目标：UAV 探索完整地图后，自动生成语义地图。

流程：

```text
探索轨迹
  RGB / Depth / Pose / Seg / State
      ↓
object detection / segmentation / tracking
      ↓
3D fusion
      ↓
object instance map
      ↓
region candidate clustering
      ↓
region/object/relation graph
      ↓
LLM/VLM 审核与命名
```

仿真内优先用真值 segmentation；真实场景再用 SAM3 / OpenMask3D / ConceptGraphs 类路线。

---

## 模块 5：Instruction & Referring Expression Generator

目标：为同一目标生成不同粒度语言。

同一个目标 `SM_DoorClassroom2` 可以生成：

```text
高层任务：
“找到二楼走廊左侧的教室门并靠近观察。”

区域引导：
“先进入教学楼二楼走廊，再寻找靠左侧墙面的教室入口。”

目标指代表达：
“那扇位于走廊左侧、靠近窗户的教室门。”

过程描述：
“无人机从走廊入口起飞，沿走廊前进，在看到左侧门牌后调整朝向并接近目标。”
```

这可以训练不同能力：

|语言类型|训练能力|
|---|---|
|高层指令|任务理解|
|区域引导|分层导航|
|目标指代|grounding|
|step-by-step|VLN 路径跟随|
|过程 caption|轨迹理解|
|失败描述|自我诊断 / failure mining|

---

## 模块 6：TaskTrace / Reasoning Label Generator

目标：给轨迹补中间推理监督。

不要让 LLM 输出自由散文式“思考过程”，而是输出结构化标签：

```json
{
  "timestep": 42,
  "current_region": "corridor_2f",
  "visible_objects": ["door_201", "door_203", "window_02"],
  "target_candidates": [
    {"object_id": "door_203", "score": 0.82, "evidence": "left side, classroom sign visible"}
  ],
  "subgoal": "approach_target",
  "next_action_intent": "move_forward_and_yaw_left",
  "memory_update": [
    "door_203 observed",
    "corridor left wall explored"
  ]
}
```

这比让模型学长篇 CoT 更安全、更可控，也更适合训练。

---

## 模块 7：Failure Reasoner / Regeneration Planner

目标：失败 episode 不是简单标注 `timeout`，而是解释为什么失败，并生成补采策略。

例子：

```json
{
  "failure_type": "target_not_visible",
  "failure_stage": "localize_target",
  "root_cause": "observe point was behind a wall, target door never entered camera frustum",
  "suggested_regeneration": {
    "change_spawn_region": "corridor_2f",
    "increase_observe_distance": 3.5,
    "require_visibility_check": true
  }
}
```

这刚好服务你文档里的 Failure Mining + Regeneration Planner 主线。AirSpark v0.7 里也明确把 QA Dashboard、Failure Mining、Regeneration Planner 作为 P3 核心交付，说明这个方向是顺着系统当前阶段走的，不是另起炉灶。

# 7. 一个完整主动数据工厂应该长这样

你想的“笔记全面的主动数据工厂”，我建议正式设计成：

```text
AirSpark AnnotationOps Pipeline

1. Scene Intake
   School / Fab / PCG / 3DGS / RealScan 场景接入

2. Semantic Inventory
   扫描 UE Actor / SemanticComponent / render pass
   输出 object_raw.json

3. Region Seeding
   手工或规则生成 building / floor / room / court / road 区域

4. Exploration Sweep
   UAV 自动探索，采 RGB-D-Seg-Pose
   生成可见性、覆盖率、候选对象

5. Object & Region Fusion
   多帧融合为 3D object map
   object-to-region 归属
   region hierarchy 补全

6. LLM Semantic Enrichment
   补 nouns / attributes / affordances / aliases / referring expressions

7. Task Generator
   根据区域图和对象图生成高层任务、子任务、成功条件

8. Expert Planner
   A*/Theta*/RRT/coverage/orbit planner 生成参考轨迹

9. TaskTrace Annotator
   给轨迹打 subgoal、memory update、target candidate、reasoning state

10. QA Validator
   检查语义一致性、可见性、路径可达、语言歧义、标注置信度

11. Dataset Builder
   输出 LeRobot / HDF5 / RLDS / 自定义 VLN-memory 格式

12. Failure Mining
   模型训练/评测失败后，回流到任务生成器补采
```

其中最核心的不是第 7 步“生成语言”，而是第 5–9 步：

> **把场景变成可查询、可推理、可训练的语义记忆。**

# 8. 评估指标怎么建？不要只评模型成功率，要评“标注系统本身”

你这个方向如果写论文，必须评三件事：

```text
1. 标注质量是否高？
2. 标注是否能提升模型推理/记忆能力？
3. 主动数据回流是否能补足模型弱点？
```

---

## 8.1 语义地图质量指标

|指标|含义|
|---|---|
|Object Detection Recall|探索后地图里找到了多少真实对象|
|Object Duplicate Rate|同一个对象是否被重复建成多个实例|
|3D Localization Error|对象 3D 中心 / bbox 误差|
|Region Assignment Accuracy|物体归属区域是否正确|
|Region Hierarchy Accuracy|篮球场属于操场、教室属于楼层这类层级是否正确|
|Relation Accuracy|left_of、inside、connected_to、near 等关系是否正确|
|Map Coverage|已探索区域 / 总可探索区域|
|Visibility Coverage|目标对象被有效观察的角度/距离覆盖率|

其中 Region Assignment Accuracy 很关键。例如：

```text
正确：
basketball_hoop → basketball_court → playground → campus

错误：
basketball_hoop → playground
```

后者虽然大区域对，但缺少中间区域，会削弱模型对细粒度区域的推理能力。

---

## 8.2 语言标注质量指标

|指标|含义|
|---|---|
|Grounding Accuracy|指令中的目标是否能唯一对应正确 object/region|
|Ambiguity Rate|一句话是否可能指向多个目标|
|Spatial Reference Accuracy|左/右/前/后/上/下/附近等空间词是否正确|
|Region Reference Accuracy|“教室门”“操场篮球场”等区域归属是否正确|
|Instruction Diversity|同一任务的语言表达多样性|
|Human Preference|人工评估自然性与可执行性|
|Planner Feasibility|按该指令是否存在可行轨迹|

注意，语言自然不是第一位，**可 grounding 才是第一位**。  
一句很漂亮但指错目标的指令是毒数据。

---

## 8.3 TaskTrace / 记忆监督质量指标

|指标|含义|
|---|---|
|Subgoal Alignment|轨迹阶段标签是否和实际位置/动作一致|
|Memory Update Precision/Recall|新发现对象/区域是否在正确时间写入记忆|
|Candidate Target Accuracy|每一步候选目标排序是否合理|
|Exploration Decision Accuracy|下一步探索区域是否合理|
|Process Completion Score|是否覆盖搜索、定位、接近、观察等必要阶段|
|Failure Explanation Accuracy|失败原因是否和真实日志一致|

这组指标是你区别于普通 VLN benchmark 的关键。

普通 benchmark 只给：

```text
instruction + path
```

你给：

```text
instruction + path + map memory + subgoal + candidate target + failure reason
```

这就有新意。

---

## 8.4 下游模型收益指标

最终要证明：

> 用 AnnotationOps 生成的标注训练，模型真的更会推理、更会记忆。

实验可以这样做：

|实验组|输入/监督|
|---|---|
|Baseline A|RGB + instruction → action|
|Baseline B|RGB + instruction + semantic target → action|
|Ours-Map|RGB + instruction + MapMemory → action|
|Ours-Trace|RGB + instruction + MapMemory + TaskTrace supervision|
|Ours-Active|加入 failure mining 回流补采|

评估：

```text
Seen scene success
Unseen object success
Unseen region success
Unseen instruction success
Target search time
Wrong-region rate
Revisit rate
Collision rate
Trajectory coverage
Failure recovery rate
```

其中我建议你重点报三个新指标：

### 1. Wrong-Region Rate

模型是否跑到了错误区域。

```text
目标：二楼教室门
错误：飞到一楼门 / 操场门 / 另一个教室门
```

这个指标能直接体现区域推理能力。

### 2. Memory Utility Gain

有无语义地图记忆带来的提升：

```text
MUG = SR_with_MapMemory - SR_without_MapMemory
```

这个能证明你的标注不是摆设。

### 3. Exploration Efficiency

目标搜索任务里：

```text
首次看到目标所需时间
首次看到目标所需路径长度
访问无关区域次数
重复访问区域次数
```

这比单纯 SR 更能体现“会不会推理搜索”。

# 10. 你这条线的论文叙事可以这样写

不要写成：

> 我们提出了一个 UE 程序化场景生成平台。

这个会撞商业 PCG/UnrealLLM/AutoUE，而且工程量吓人。

应该写成：

> 我们提出一个面向低空导航—操作任务的 **LLM-assisted Semantic Annotation and Map-Memory Data Factory**。它能从仿真或真实探索轨迹中构建层级区域图、对象图、关系图和任务过程轨迹标注，并生成可训练 VLN/VLA 推理记忆能力的数据。

核心贡献可以是：

1. **Hierarchical Low-altitude Semantic Map**  
    面向 UAV 的区域—对象—关系层级图，支持建筑、楼层、教室、操场、篮球场、道路、外立面等多尺度区域建模。
    
2. **LLM-assisted AnnotationOps**  
    利用 LLM/VLM 自动补全对象语义、区域归属、空间关系、自然语言指令和任务过程标签。
    
3. **Map-memory Supervision for VLN/VLA**  
    不只提供轨迹 imitation 标签，还提供探索记忆、候选目标、子目标阶段、失败原因等中间监督。
    
4. **Active Failure-driven Data Flywheel**  
    从模型失败中自动定位错误区域、错误目标、不可达路径、观察点不合理等原因，并驱动补采。
    
5. **Navigate-then-Manipulate Extension**  
    后续接入接触操作，让语义地图不仅服务导航，还服务接近、对齐、操作和恢复。
    

这比“我做了更多场景”更扎实。

# 11. 最终判断

你的方向调整非常合理，但建议把“LLM 辅助标注”再升级成一个明确系统：

```text
AirSpark AnnotationOps =
  Semantic Patch
+ RegionGraph
+ ObjectGraph
+ RelationGraph
+ Exploration MapMemory
+ TaskTrace
+ Language Generator
+ Failure Mining
+ Active Regeneration
```

短期不要幻想全自动标注所有区域。最稳路线是：

```text
P3.1：利用 UE 真值 + LLM 补全对象语义
P3.2：人工粗标区域层级，自动 object-to-region
P3.3：生成语言、子任务、地图记忆、推理标签
P3.4：用这些数据训练/评测 VLN memory 模型
P4：再引入 SAM3 / 3DGS / 真实探索做自动语义地图
```

一句话概括：

> **AirSpark 的重点不该是造世界，而是把已有世界变成“可理解、可记忆、可推理、可训练”的低空任务数据。**

这个方向更轻、更准，也更像你真正想发的东西。
