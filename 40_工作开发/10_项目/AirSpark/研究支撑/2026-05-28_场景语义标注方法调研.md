# 场景语义标注方法调研

> 面向机器人导航与具身智能的场景语义标注方案综述，覆盖 3D 场景图构建、层级区域建模、LLM 语言标注生成、任务驱动语义压缩四个方向。

---

## 一、核心结论

当前 LLM/VLM 场景标注的主流范式不是"LLM 直接看世界然后全自动标完"，而是：

> **分割/检测模型给 object proposal → 深度/位姿融合成 3D 实例 → 几何规则建区域/关系 → LLM/VLM 补语义、命名、语言和任务描述。**

各模块职责划分：

```
分割模型（SAM3 / GroundedSAM / YOLO-seg）  — object proposal
深度 + 相机位姿                              — 3D 融合
几何规则 / 拓扑分析                          — 区域划分 + 关系推断
LLM / VLM                                  — 语义命名 + 语言生成 + 任务推理
QA / 规则 / 人工抽检                         — 防幻觉 + 质量验证
```

---

## 二、路线 A：2D Foundation Model → 3D Object Graph

从 RGB-D 或多视角图像出发，利用 2D 分割/特征模型构建开放词汇 3D 对象图。适用于**无预设标签的真实/未知场景**。

### 通用流程

```
RGB-D / 多视角图像
    ↓
2D segmentation / detector / CLIP feature
    ↓
depth + camera pose 投影到 3D
    ↓
多视角融合成 object instance
    ↓
open-vocabulary 3D object map / scene graph
```

### ConceptGraphs

**定位**：open-vocabulary 3D scene graph，面向 perception and planning。

核心贡献：
- 利用 2D foundation model（CLIP、DINO、SAM）提取 per-frame object proposals 和特征
- 通过 depth + pose 投影到 3D，多视角关联融合成 object instance
- 构建 object-centric graph，支持开放词汇查询和空间关系推理

适用场景：真实探索轨迹 → 语义地图；无 actor 级标签的第三方场景。

> 仓库：[concept-graphs/concept-graphs](https://github.com/concept-graphs/concept-graphs)

### OpenMask3D

**定位**：open-vocabulary 3D instance segmentation。

核心贡献：
- class-agnostic 3D mask 提取（不依赖预定义类别）
- 多视角 CLIP 图像特征聚合到 3D mask
- 支持自由文本查询 3D 实例

适用场景：3DGS / 真实扫描 / 点云场景，没有 actor 级标签时生成 3D object proposals。

> 仓库：[OpenMask3D](https://openmask3d.github.io)

### Open3DIS

**定位**：2D mask guidance 做 3D 开放词汇实例分割。

与 OpenMask3D 属于同类备选，强调用 2D mask 引导 3D 分割，适合真实点云/扫描场景补 object masks。

> 仓库：[VinAIResearch/Open3DIS](https://github.com/VinAIResearch/Open3DIS)

### SAM3 / SAM3.1

**定位**：支持文本 prompt 或视觉 prompt 的图像/视频分割与跟踪。

SAM3 相比 SAM/SAM2 的升级：
- 支持 open-vocabulary text prompt 和 visual prompt
- 可在图像和视频中检测、分割、跟踪匹配对象
- SAM3.1 重点提升视频处理效率，适合长视频/探索轨迹

**使用边界**：SAM3 给的是 2D/视频 object mask，不会自动给可靠的 3D 区域层级。它是感知工具，不是数据工厂范式。

适用场景：
- 真实/未知场景，没有预设 actor ID 和语义标签
- 第三方场景资产名混乱，需要自动发现对象
- 低置信度对象的辅助确认

> 仓库：[facebookresearch/sam3](https://github.com/facebookresearch/sam3)

## 三、路线 B：层级 3D Scene Graph

构建 floor / room / object 层级结构，支持语言导航和空间推理。这是最接近"可训练 VLN 记忆"的方向。

### HOV-SG

**定位**：Hierarchical Open-Vocabulary 3D Scene Graphs for Language-Grounded Robot Navigation（RSS 2024）。

核心贡献：
- 先构建开放词汇 3D segment-level map
- 再组织成 floor → room → object 层级 scene graph
- 每层都有开放词汇特征，支持自然语言查询

典型指令示例：
```
"go to the toilet in the bathroom on floor 2"
"go to the blue sofa in the corridor on floor 2"
"navigate to the box in the seminar room on floor 2"
```

支持 large-scale、multi-story environments 的 language-grounded robot navigation。

关键设计启示：
1. RegionGraph 必须层级化（campus → building → floor → corridor/classroom）
2. object 不应孤立存在，要挂到 room/floor/building
3. 查询不是"找一个门"，而是"找二楼走廊里的 A101 门"
4. 语言导航最好依赖 region hierarchy，而不是平面 object list

> 仓库：[hovsg/HOV-SG](https://github.com/hovsg/HOV-SG)

### OpenGraph

**定位**：大尺度户外开放词汇层级图（RAL 2024）。

核心贡献：
- 从图像提取实例和 captions
- 投影到 LiDAR 点云做 3D incremental object-centric mapping
- 基于 lane graph connectivity 切分环境，构建层级图

流程：
```
图像实例 + caption
    ↓
投影到 LiDAR/点云
    ↓
object-centric map
    ↓
道路/车道/区域连通图
    ↓
hierarchical outdoor graph
```

与 HOV-SG 的区别：HOV-SG 偏室内多层建筑，OpenGraph 偏大尺度户外（道路、广场、园区）。城市低空/校园场景两者都有参考价值。

> 仓库：[BIT-DYN/OpenGraph](https://github.com/BIT-DYN/OpenGraph)

### SG-Nav

**定位**：Online 3D Scene Graph Prompting for LLM-based Zero-shot Object Navigation。

核心思路：把观测场景表示成 3D scene graph，再 prompt LLM 做导航推理。只给 LLM nearby objects 的文本缺少足够 scene context，3D scene graph 能更好保留环境信息并利用 LLM 推理能力。

对 MapMemory 设计的支撑：说明不能只做 object list，必须保留区域层级和空间关系，才能让 LLM 做有效的导航推理。

> 项目页：[SG-Nav](https://bagh2178.github.io/SG-Nav)

## 四、路线 C：Task-Driven 语义压缩

不追求全量标注，根据任务需求保留有价值的对象和区域。

### Clio

**定位**：Real-time Task-Driven Open-Set 3D Scene Graphs（MIT-SPARK）。

核心思路：
- 根据自然语言任务，把 object primitives 聚成任务相关的 semantic regions
- 只保留任务相关语义概念，使图更紧凑
- 提升任务执行准确性

示例：同一个空间，"取咖啡"任务只保留 Kitchenette 区域相关对象；"开会"任务只保留 Workspace 相关对象。

关键启示：

> 不需要把场景里每一块墙、每个地砖都标到极致。更重要的是标出对任务有用的对象和区域。

对象价值分级示例：
```
高价值：门、窗、标志牌、楼梯、走廊、教室、操场、篮球场、障碍物、目标设备
低价值：普通墙面小块、重复地砖、无任务意义装饰物
```

建议在 semantic patch 中加入：
```json
{
  "task_relevance": "high | medium | low",
  "task_roles": ["navigation_landmark", "target_candidate", "region_boundary", "obstacle", "background"]
}
```

> 仓库：[MIT-SPARK/Clio](https://github.com/MIT-SPARK/Clio)

## 五、路线 D：LLM 语言/任务标注生成

利用已有场景结构生成语言和训练数据，不重点做分割。

### NavRAG

**定位**：LLM 构建 hierarchical scene description tree + RAG 生成用户需求导航指令。

核心贡献：
- 从 global layout 到 local details 构建层级 scene tree
- 模拟不同用户需求（安防巡检员、配送员、应急响应员、维修工程师）
- 用 LLM + RAG 生成多样化、贴合应用场景的导航指令
- 标注了 861 个场景上的 200 多万条导航指令

关键结论：**语言指令不应该只从轨迹模板生成，而应该从场景树 + 用户需求生成。**

示例（同一目标，不同角色）：
```
安防巡检员：  "巡检二楼走廊尽头的消防栓和旁边教室门。"
维修人员：    "找到 A101 教室门旁边的设备箱并靠近观察。"
低空无人机：  "沿教学楼外立面飞行，检查二楼窗户区域。"
```

> 仓库：[MrZihan/NavRAG](https://github.com/MrZihan/NavRAG)

### MAPInstructor

**定位**：Scene Map-based Prompt Tuning for Navigation Instruction Generation（CVPR 2025）。

核心贡献：
- 把 scene map 作为 latent features 引入 LLM 的 prompt tuning
- 两阶段生成：landmark prediction → instruction generation
- 加入 landmark uncertainty assessment 减少幻觉

关键结论：生成导航指令时，不应该只给 LLM 一串对象名，应该给它 region path、关键 landmark、拓扑关系、可见性、路径转折点。

对比：
```
不好：
  目标：A101门 / 起点：大厅

更好：
  起点区域：大厅
  路径区域：大厅 → 一楼走廊
  关键 landmark：楼梯口、厕所标志、A101门牌
  转向事件：进入走廊后左转
  可见性：门牌在 5m 内可见
```

> 论文：[CVPR 2025 Open Access](https://openaccess.thecvf.com/content/CVPR2025/papers/Fan_Scene_Map-based_Prompt_Tuning_for_Navigation_Instruction_Generation_CVPR_2025_paper.pdf)

### 3D-GRAND

**定位**：大规模 3D-text dense grounding 数据集（CVPR 2025）。

规模：40,087 个 household scenes，6.2 million densely-grounded scene-language instructions。

核心贡献：
- 每条语言都 ground 到具体 object / region / relation
- 通过 dense grounding 提升 3D-LLM grounding 能力、减少 hallucination
- 提出 3D-POPE 评估 3D-LLM 幻觉

数据格式示例：
```json
{
  "instruction": "找到A101教室门",
  "grounding": {
    "target_object": "door_A101",
    "target_region": "corridor_1f",
    "mentioned_region": "classroom_A101",
    "relations": [
      ["door_A101", "connects", "corridor_1f"],
      ["door_A101", "belongs_to", "classroom_A101_entrance"]
    ]
  }
}
```

关键结论：不要只保存一句 instruction，每条语言都要能 ground 到 object/region/relation，这是训练 VLN 推理记忆能力的关键。

> 仓库：[sled-group/3D-GRAND](https://github.com/sled-group/3D-GRAND)

### Holodeck

**定位**：Language Guided Generation of 3D Embodied Environments。

核心思路：LLM 生成物体之间的空间关系约束，再由优化器/规则系统落地几何布局，而不是直接输出最终几何。

借鉴点：LLM 不判断精确几何，LLM 生成语义/关系候选，几何规则/真值验证。这和 LLM 标注的通用原则一致。

> 项目页：[Holodeck](https://yueyang1996.github.io/holodeck)

## 六、方案对照表

| 外部工作 | 核心方法 | 适用场景 | 主要局限 |
|----------|----------|----------|----------|
| SAM3 | 文本/视觉 prompt 图像/视频分割 | 真实/未知场景 object discovery | 只给 2D mask，不给 3D 区域层级 |
| ConceptGraphs | 2D foundation model + RGB-D → 3D scene graph | 真实探索轨迹转 ObjectGraph | 需要 RGB-D + 相机位姿 |
| OpenMask3D | 3D open-vocabulary instance segmentation | 点云/扫描场景 object proposals | 不负责区域层级和任务语言 |
| Open3DIS | 2D mask guidance 3D OV instance segmentation | 真实点云场景 | 同 OpenMask3D |
| HOV-SG | floor-room-object 层级开放词汇 3D scene graph | 室内多层建筑导航 | 偏室内，户外支持有限 |
| OpenGraph | 大尺度户外开放词汇层级图 | 城市/园区/道路场景 | 依赖 LiDAR |
| Clio | task-driven open-set 3D scene graph | 任务相关语义压缩 | 需要明确任务定义 |
| SG-Nav | 3D scene graph prompt LLM 做 zero-shot 导航 | 在线导航推理 | 实时性要求高 |
| NavRAG | scene tree + RAG + LLM 生成用户需求指令 | 多样化导航指令生成 | 依赖高质量 scene tree |
| MAPInstructor | scene map-based prompt tuning 指令生成 | 地图驱动语言生成 | 需要 topological map |
| 3D-GRAND | dense 3D-language grounding 数据集 | 训练 3D-LLM grounding 能力 | household 场景，非户外 |
| Holodeck | LLM 生成空间关系约束 + 优化器落地 | 室内场景生成 | 生成场景，非标注 |

## 七、推荐阅读优先级

按价值从高到低：

1. **HOV-SG** — 层级区域图设计的直接参考，看 floor-room-object 如何组织并服务 language-grounded navigation
2. **ConceptGraphs** — 真实场景 ObjectGraph 构建的标准范式，看 2D foundation model → 3D object graph 全流程
3. **OpenGraph** — 大尺度户外层级图，城市低空/道路/校园场景更接近
4. **Clio** — task-driven 语义压缩，避免全量无脑标注
5. **NavRAG** — scene tree + RAG + LLM 生成多样化导航指令
6. **3D-GRAND** — dense grounding 数据格式和 hallucination 评估方法
7. **MAPInstructor** — 地图上下文参与指令生成的具体实现
8. **SAM3** — 了解工具能力和 API，按需使用

## 八、关键设计原则（综合）

**1. 分工明确，不要让 LLM 做几何判断**
```
几何/规则 → 区域边界、对象归属、空间关系
LLM/VLM  → 语义命名、语言生成、任务推理
```

**2. 区域必须层级化，不能扁平分类**
```
campus → building → floor → corridor/classroom/restroom
campus → playground → basketball_court / running_track
```

**3. 语言必须 ground 到 object/region/relation**

每条 instruction 都要能唯一对应正确的 object 和 region，可 grounding 优先于语言自然。

**4. 标注要 task-driven，不要全量冗余**

根据任务类型（导航/巡检/操作）保留有价值的对象，给每个对象标 `task_relevance` 和 `task_roles`。

**5. 指令生成必须输入地图上下文**

不能只给 LLM 目标名称，要给 region path、关键 landmark、拓扑关系、可见性、路径转折点。

**6. 仿真场景优先用真值，真实/未知场景再用感知模型**

有 actor 级标签和精确 3D 坐标时，不需要用 SAM3/ConceptGraphs 重新感知。感知模型是真实/无标签场景的前端，不是仿真场景的替代。
