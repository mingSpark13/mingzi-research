## 🔍 核心发现：UniDex 其实是纯端到端 VLA

### UniDex 的真实架构

```
输入：pointcloud + language instruction + proprioception（灵巧手关节状态）
     ↓
UniDex-VLA（3D VLA，端到端训练）
     ↓
输出：FAAS 动作空间的 action chunks（7-24 DoF）
```

**重要澄清**：UniDex 是 **纯端到端 VLA**，不是层次化架构！

它和 RT-2 / OpenVLA 一样，都是 VLM 直接输出 action，只是：
1. 用 **pointcloud** 而不是 RGB 图像作为视觉输入
2. 用 **FAAS** 统一动作空间，支持跨手型迁移

### FAAS 的本质

**FAAS = Function-Actuator-Aligned Space**

把「功能相似的执行器」映射到统一坐标：
- 比如：不同灵巧手的拇指虽然关节数量/角度不同，但都是「拇指功能」
- FAAS 把这些「功能相似的执行器」映射到共享坐标
- 训练时用 FAAS 统一表示，推理时可以映射回具体手型

**这不是层次化分解，而是动作空间的归一化表示。**

主人说：
> 纯端到端 VLA 不适合空中精细操作
> VLM 给出模糊意图 → 其他模块优化为安全精细高频操作

**UniDex 是纯端到端 VLA**，它预测的就是精细操作（灵巧手 7-24 DoF），不是「意图」。

这意味着：**UniDex 没有解决主人的核心关切**——它没有展示 VLM 和低层控制如何分离。

## 💡 花火的新发现

### 问题：主人想要的架构是什么？

根据主人纠正，真正需要的架构是：

```
VLM（慢，秒级）→ 模糊意图/任务分解
      ↓
专门的操作模块（快，毫秒级）→ 精细动作序列 + 安全约束
      ↓
电机控制
```

### 花火发现的空白

**现有工作要么是：**
1. **纯端到端 VLA**（RT-2, OpenVLA, π0, UniDex）— VLM 直接输出 action
2. **纯导航混合架构**（OnFly）— VLM 只做裁判，轨迹规划器生成控制

**缺少的是：**
3. **层次化 VLA 操作架构** — VLM 做任务规划 + 低层精细操作控制

这不是简单的「VLM 输出 action 了」，而是**把 VLM 和低层控制解耦**，让 VLM 做「该做什么」（意图），低层 NN 做「怎么做」（精细动作）。

## 🎯 花火的新判断：潜在的正确方向

### 方向X（花火重新归纳）：层次化「意图→精细操作」空中操作系统

**核心思想**：
- 高层：VLM/LLM 接收语言指令 + 点云观测 → 输出「任务意图」（语义层，如"拿起杯子放到桌子左侧"）
- 中层：意图解析器把「意图」分解为「操作序列」（如"夹爪张开→移动到杯子上方→下降→夹爪闭合→提升"）
- 低层：每步的精细执行（毫秒级 PID / 阻抗控制）

**创新点**：
1. 首次把「层次化意图→操作」架构用于空中精细操作
2. VLM 只负责「意图层」，不直接输出精细动作
3. 中层操作序列可解释、可调试
4. 低层控制独立，可换策略

**架构图**：
```
语言指令 + 点云观测
        ↓
【高层】VLM/LLM → 任务意图（语义层）
        ↓
【中层】意图解析器 → 操作序列（夹爪开/合 + 末端执行器轨迹）
        ↓
【低层】精细执行控制器 → 电机指令（毫秒级，PID/阻抗）
        ↓
电机
```

### 方向Y（相关但次要）：OnFly 扩展到操作

OnFly 的「语义-几何验证」思路可以迁移：
- VLM 做操作成功/失败的语义判断
- 几何方法做碰撞检测和安全约束
- 但 OnFly 本身只有导航，没有操作层
