# D05: 数据飞轮 研究报告

> 最后更新:2026-04-18 | 成熟度:🟡中期（R791把数据飞轮从“组件清单”收束到 E1-E5 最小闭环与四条正式实验轴）
> 状态：🟡 推进中

## 一、研究背景与动机

具身智能策略训练的核心瓶颈是数据。真实采集每条轨迹成本高(人工示教+环境重置),且难以覆盖足够多的任务/物体/场景变体。MolmoBot 证明 180 万条仿真轨迹可实现 79.2% 的 zero-shot 真机成功率(vs π0.5 的 39.2%),说明大规模仿真数据路线的上限足够高。问题是:如何自动化地构建这条数据管线?

## 二、相关工作梳理

### 2.1 资产生成(真实→仿真)
- **PAWS** - 从第一视角视频挖掘 articulation
- **Pandora** - 人类探索视频 → articulated scene graph
- **SIMART** - 静态 mesh → sim-ready articulated asset
- **URDF-Anything+** - 视觉观测 → 可执行 URDF
- **ExoGS** - 4D Real-to-Sim-to-Real,直接捕获动态交互轨迹,并把机器人/物体/环境重建成可编辑 3DGS 资产

### 2.2 轨迹生成
- **V-Dreamer** - LLM + 3D 生成 → 场景 + 轨迹自动合成
- **Physics-Driven** - 少量人类示教 + 轨迹优化 → 跨载体合成轨迹
- **SoftMimicGen** - 可变形物体自动数据生成
- **AGT-World** - Affordance graph 自进化任务生成

### 2.3 闭环采集
- **RADAR** - 语义规划 + 自主执行 + 因果环境重置
- **EgoSim** - 3D 状态持久化闭环模拟器
- **IWS** - 交互式世界模拟器,长时稳定交互

### 2.3+ 🆕 R781 新增：少样本演示扩数
- **AffordGen** (2604.10579) — 用 affordance correspondence 把少量人类示教扩成跨新物体类别、全 6D pose 的大量操作轨迹。它最值得 D05 保留的不是“又一种数据增强”，而是明确给出一条 **少量真实种子示教 → affordance 对齐扩数 → 零样本泛化** 的数据链路，正好补在纯仿真生成与元数据自标注之间。

### 2.4 数据工厂基础设施
- **Nimbus** (2601.21449) — 统一 synthetic data framework，把轨迹规划、渲染、存储拆成异步阶段，用动态调度、全局负载均衡和容错机制把 CPU/GPU/I/O 资源打满，端到端吞吐提升 2-3 倍。它最值得借鉴的不是"再来一个仿真器"，而是 **数据工厂编排层**：后续 UE / AirSim 管线应把任务生成、渲染录制、落盘校验拆开，避免单进程串行采集拖死吞吐。

### 2.5 Gaussian Splatting 数据合成（AirVLA — 2026-04-16 R743 新增）
**AirVLA (2603.25038)** 用 Gaussian Splatting pipeline 合成 navigation 训练数据，解决 aerial 操作数据稀缺问题。

**关键发现**：
- 仅遥操数据 → 导航 81% 成功率
- 遥操 + Gaussian Splatting 合成数据 → 100% 成功率
- **合成数据是性能跃升的关键使能器**

**对 D05 的意义**：
- Gaussian Splatting 比传统仿真器更易获取真实感场景（从少量真实观测自动重建）
- 可作为「世界模型替代仿真器」路线的具体技术方案
- **可落地方向**：为龙虾项目采集少量真实空中操作轨迹 → 3DGS 重建 → 大规模合成数据扩展
- **Robot-Powered Data Flywheel / Scanford** (2511.19647) - 展示了真实部署中的 **任务执行 + 自动采集 + 环境元数据自标注 + 模型回灌** 闭环。关键启发不是"机器人顺手采数",而是要主动寻找 **catalog / inventory / map / log / task database** 这类环境自带的结构化监督源,把它们接成低成本 self-labeling 管线,减少人工标注依赖。

### 2.5 大规模验证
- **MolmoBot** - 180 万仿真轨迹 → zero-shot 79.2%

### 2.6 完整链路
```
PAWS/Pandora → SIMART/URDF-Anything+ → V-Dreamer/AGT-World → RADAR/EgoSim → MolmoBot
(结构发现)    (资产化)              (轨迹生成)          (闭环采集)   (策略训练)
```

### 2.7 Gap
1. 整条链路尚无人端到端打通(各环节独立验证)
2. **空中操作场景的数据工厂完全空白**
3. 世界模型能否直接替代仿真器刷数据(IWS 初探,未验证规模效应)
4. ExoGS 证明"动态交互一起采"可行,但仍偏机械臂近场操作,尚未覆盖空中平台的视角变化与飞行动力学
5. **frontier video model 可做高层轨迹提议,但低层精控仍不足**--Veo-Act 说明视频模型适合作为轨迹提议器/数据标注器,而不是直接替代执行控制器
6. **缺少面向长期高吞吐运行的数据工厂编排层**--Nimbus 证明吞吐瓶颈常在调度、容错与 I/O,而不只是"生成器够不够强"
7. **缺少面向真实部署的低成本自标注链路**--Scanford 证明现实环境往往自带 catalog / inventory / 日志等监督源,但当前 D05 还没把这类 environment metadata 作为正式数据工厂组件纳入设计
8. **少样本真实示教如何高保真扩成跨物体操作数据仍未独立建模**--AffordGen 表明 affordance correspondence 是一条比“纯视频生成”更稳的几何/语义对齐扩数路线,值得单列为数据工厂正式组件

## 三、我们的创新方向

### 3.1 核心创新点（收束版）

1. **C1: 空中操作数据工厂**
   - 把 `资产生成 → 任务生成 → 轨迹采集 → 回灌训练` 整条链路适配到无人机/空中机械臂场景
   - 不再只产地面机械臂式静态抓取数据，而是显式覆盖视角剧变、飞行动力学扰动与导航后操作

2. **C2: 混合数据飞轮，不押单一路线**
   - 同时保留四条增益链：
     - `程序化仿真扩数`（F-ACIL / Nimbus）
     - `少量真实锚点扩数`（AffordGen / AffordSim）
     - `环境元数据自标注`（Scanford）
     - `策略自举回收`（DexFlyWheel）
   - 目标不是证明哪一条“唯一正确”，而是找出主人当前资源下最划算的组合

3. **C3: 跨载体 paired data 自动生成**
   - 直接服务 D04，把“同任务-不同载体”的轨迹对自动产出来
   - 优先支持 `地面机械臂 → 空中机械臂`、`human geometry → robot execution` 两类映射

4. **C4: 世界模型/视频模型做高层数据引擎，不做低层控制替代**
   - 世界模型、Veo-Act、Action Images 等更适合做 `任务级轨迹提议 / 标签补全 / 候选rollout生成`
   - 低层控制与可执行性筛选仍交给 VLA / 控制器 / verifier

5. **C5: 几何一致增强 + 失败恢复数据并重**
   - 一方面用 RoboTransfer 类方法做背景替换、目标替换、多视角一致扩增
   - 另一方面主动收集 near-failure recovery data，而不是只积累干净成功轨迹

### 3.2 拟定方法框架

```text
真实少量锚点 / 人类HOI / 环境元数据 / 程序化仿真
                │
                ▼
      统一数据协议层（object / action / environment / disturbance / failure）
                │
      ┌─────────┼─────────┐
      ▼         ▼         ▼
  资产生成   任务生成   自标注/回收
 (PAWS等)   (V-Dreamer) (Scanford/DexFlyWheel)
      └─────────┬─────────┘
                ▼
      轨迹扩数与增强层
   (AffordGen / AffordSim / RoboTransfer)
                │
                ▼
      执行闭环与状态持久化
      (RADAR / EgoSim / IWS)
                │
                ▼
      策略训练与回灌评估
   (MolmoBot / VLA / D04 paired transfer)
```

### 3.3 四条正式实验轴

| 实验轴 | 核心问题 | 代表路线 | D05 中的定位 |
|---|---|---|---|
| **轴A: 因子化协议** | 数据先按什么结构组织，才能高效覆盖组合角落 | F-ACIL / Nimbus | 整个数据工厂的底座 |
| **轴B: 少量真实锚点扩数** | 少量真机/人类示教能否放大成高价值数据池 | AffordGen / RoboWheel | 降低真机冷启动成本 |
| **轴C: 自标注与策略自举** | 是否能在真实执行中持续自动长出新样本 | Scanford / DexFlyWheel | 形成长期飞轮闭环 |
| **轴D: 失败恢复与可执行性** | 数据工厂是否真正产出“难而有用”的样本 | TAMEn / EgoSim | 决定部署稳健性上限 |

### 3.4 与现有方法的关键差异

| 对比维度 | 纯程序化生成 | 纯真实示教扩数 | 纯元数据自标注 | 我们的方法 |
|---|---|---|---|---|
| 冷启动成本 | 低 | 高 | 中 | 低到中 |
| 物理真实性 | 中 | 高 | 中 | 用少量真实锚点补高 |
| 多样性 | 高 | 中 | 中 | 高 |
| 跨物体泛化 | 取决于随机化质量 | 取决于示教覆盖 | 取决于环境结构 | 用 affordance + factorization 强化 |
| 失败恢复样本 | 弱 | 中 | 弱 | 显式纳入正式产物 |
| 对主人当前可落地性 | 易开工但易飘 | 太贵 | 依赖场景 | 最均衡 |

## 四、实验设计

### 4.1 最小系统闭环

先不追“全链路全自动”，而是做一个能回答关键问题的最小闭环：

1. **真实少量锚点采集**
   - 选 2 到 3 类目标物体，采少量真实或高保真示教
   - 每类只要够学到接触位与 pre-grasp 几何即可

2. **仿真/增强扩数**
   - 用 AffordGen 路线做姿态、物体类别、视角扩展
   - 用 RoboTransfer / 程序化扰动补背景、视角、载体差异

3. **闭环执行与失败回收**
   - 在 EgoSim / UE/AirSim 内执行
   - 单独收集 near-failure recovery data，不和成功轨迹混为一类

4. **训练与回灌**
   - 把扩增数据、失败恢复数据、少量真实锚点分桶训练
   - 评估是否真提升 D04 跨载体迁移或 D06/D07 下游任务成功率

### 4.2 数据链路对照表（R791推进）

| 实验编号 | 数据来源 | 是否有真实锚点 | 是否有元数据自标注 | 是否收集失败恢复 | 目标 |
|---|---|---|---|---|---|
| **E1** | 纯程序化仿真 | 否 | 否 | 否 | 纯低成本 baseline |
| **E2** | 程序化仿真 + 少量真实锚点 | 是 | 否 | 否 | 验证 AffordGen 路线是否值得保留 |
| **E3** | E2 + 环境元数据自标注 | 是 | 是 | 否 | 验证真实部署自标注收益 |
| **E4** | E2 + 失败恢复数据 | 是 | 否 | 是 | 验证 near-failure 样本对稳健性的真实价值 |
| **E5（ours）** | 真实锚点 + 仿真扩数 + 自标注 + 失败恢复 | 是 | 是 | 是 | 完整 D05 最小飞轮 |

### 4.3 评价指标

- **SR**：下游任务成功率
- **Anchor Efficiency**：每增加 1 条真实锚点数据带来的成功率提升
- **Recovery Gain**：加入失败恢复数据后的稳健性提升
- **Coverage**：object / action / environment / disturbance / failure 五类标签的组合覆盖率
- **Replayability**：采到的数据能否稳定回放、复现与再次利用
- **Transfer Utility**：数据是否真实提升 D04 / D06 / D07 的下游实验，而不只是离线指标变好

### 4.3+ 首轮飞轮准入门与样本记账协议（R821 新增）

D05 当前最缺的不是再加一个数据源，而是先把“哪些样本值得进飞轮”写成统一协议。首轮建议把所有样本统一记到四个桶里：

| 桶 | 定义 | 最小字段 | 进入下轮训练的默认策略 |
|---|---|---|---|
| **B1 成功样本** | 成功完成任务且轨迹可稳定回放 | object/action/environment/disturbance/outcome | 直接进入主训练集 |
| **B2 近失败样本** | 未完成任务但距离成功一步之遥，如抓偏、对准失败、局部碰撞后退出 | failure_type/failure_phase/last_safe_state | 优先保留给恢复训练与 verifier |
| **B3 恢复样本** | 从失败状态被人工或策略拉回成功 | recovery_source/recovery_steps/recovery_cost | 单独计入 recovery 桶，不与 B1 混合 |
| **B4 自标注样本** | 由目录/地图/日志/任务配置等环境元数据自动附上弱标签 | metadata_source/label_confidence/schema_version | 先作为辅训练数据，需抽样审计 |

基于这四个桶，首轮只放行能同时满足以下门槛的数据链路：

- **G0 Replayable**：同一轨迹至少可稳定回放 2 次，不因资产/初始化漂移失真
- **G1 Labeled**：五类因子字段至少填满 `object / action / environment / outcome`
- **G2 Useful Failure**：失败样本里至少 30% 能明确归到可行动的 failure type，而不是只有“失败”
- **G3 Transfer-linked**：至少有一项下游指标能映射到 D04 / D06 / D07，例如跨载体迁移、导航恢复或控制稳健性

核心目的不是一开始就把飞轮做大，而是先避免“采了很多、但没有可回放失败样本、没有 schema、也接不上下游”的伪闭环。

### 4.4 消融实验设计

1. **Affordance 扩数是否真比纯随机化强**
   - E1 vs E2
2. **环境元数据自标注是否真值钱**
   - E2 vs E3
3. **失败恢复数据是否比再加成功轨迹更有效**
   - E2 vs E4
4. **因子化 schema 是否减少无效数据堆积**
   - 有/无 `object-action-environment-disturbance-failure` 五类标签
5. **真实锚点比例消融**
   - 10 / 20 / 50 / 100 条真实锚点，比较收益拐点

### 4.4+ 首轮执行顺序（R821 新增）

1. **先过 E0 烟测**：验证 `真实锚点 10-20 条 → 扩数 → 回放 → 训练` 能完整跑通
2. **再比 E1 vs E2**：先回答“少量真实锚点扩数值不值”
3. **然后只开一条增强支线**：优先在 `E3 自标注` 和 `E4 失败恢复` 里二选一，避免变量过多
4. **最后再合并到 E5**：只有当 replayability 与 bucket 记账稳定后，才进入完整飞轮

这样做的原因是，D05 当前最大的风险不是模型不够强，而是链路一上来过长，最后无法判断收益到底来自真实锚点、schema、自标注，还是失败恢复样本。

### 4.5 硬件与实现优先级

| 模块 | 最低配置 | 推荐配置 | 备注 |
|---|---|---|---|
| UE/AirSim 数据采集 | 单张 3090 | 4090 / 多实例 | 先保证可并行录制 |
| 资产/视频增强 | 单卡即可 | 单卡 + 大容量SSD | 重点是 I/O 稳定 |
| 训练与评测 | 单张 24GB GPU | 48GB 或多卡 | 不需要一开始就追大模型 |
| 日志与元数据 | 充足磁盘 | NVMe + 结构化索引 | 这部分决定飞轮能否持续 |

**当前建议**：D05 首轮不要把算力砸在更大模型上，先把 `数据协议 + 异步编排 + 失败恢复回收` 三件事做稳。
