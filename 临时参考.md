架构定位

这是一个 AirSpark 低空无人机 VLA 仿真与数据生产平台，不是单纯 AirSim demo。当前主线是：

UE 5.7 + AirSim fork + URLab/MuJoCoBackend + School 地图 + RPC 41451 + UE 内部批采集 + VLA raw collection

核心判断来自 AirSpark-Plugin-Architecture.md (line 38) 和 ROADMAP.md (line 15)。

当前分层大致是：


Unreal/airworld：主 UE 工程壳。

Plugins/AirSim：真正的核心平台层，已深度 fork AirSim。

Plugins/AirSim/Source/AirSpark：业务主线，已有 12 个子模块，包括 Batch、Mission、Navigation、Collection、Episode、Randomization、Runtime、Semantic 等，见 AirSpark-Plugin-Architecture.md (line 11)。

PythonClient：RPC 调试、烟测、外部控制入口，不应再作为高带宽数据面。

tools/airspark_dataset_tools：离线 validator、LeRobot-like converter、evaluator、randomization dry-run。

Matlab、Blender、3DGS/MLSLabs：控制实验、资产、视觉重建旁路，不是当前 VLA 主链路。


当前成熟度

P2“UE 内部批采集”已经基本成型：BatchController 状态机、MissionDirector、WaypointExecutor、JSONL writer、manifest、image capture queue、离线 validator/converter 都有实现。模块边界文档也明确要求避免 MissionDirector、BatchController、Collector 继续膨胀，见 Module-Boundaries.md (line 3)。

我跑了离线工具：

没有跑 UE build/Play，因为当前 URLab 子模块未初始化。

主要缺陷/风险


当前工作区无法可靠构建 UE 主线

Unreal/airworld/Plugins/URLab 是空目录，git submodule status 显示未初始化；但 .uproject 启用了 UnrealRoboticsLab，AirSim.Build.cs 也依赖 URLab。这是第一优先级的复现性问题。



文档事实源不一致

ROADMAP.md 说 sensor capture queue、trajectory optimizer 已完成，但 README.md (line 184)、Programmatic-Collection-Usage.md (line 19)、tools/airspark_batch_runner/README.md (line 113) 仍写着 stub / placeholder / not implemented。团队继续开发时会被旧文档误导。



max_retries_per_episode 只进 schema，没有被 BatchController 消费

ROADMAP 已承认该问题，见 ROADMAP.md (line 94)。这会影响大批量采集的自动恢复能力。



多 drone 并行仍未架构闭环

RuntimeProfile 仍有全局 override 和 single-world 假设，见 AirSparkRuntimeProfile.cpp (line 15)。后续多实例、多车、多采集 worker 会踩到这个边界。



导航规划仍有硬缺口

AirSparkOccupancyGridPlanner 里仍有 “Occupancy grid search is not implemented yet”，见 AirSparkOccupancyGridPlanner.cpp (line 376)。复杂室内/低空遮挡场景会成为数据覆盖率瓶颈。



若干类已经偏大

AirSparkWaypointExecutorComponent.cpp 约 1651 行，SceneDebugSubsystem.cpp 约 974 行，WaypointPlanner.cpp 约 681 行。短期可接受，但继续堆功能会让线程生命周期、采集状态、飞控命令和 QA 日志互相缠绕。



自动化验证不足

.github 当前为空且 .gitignore 里忽略 .github。Python 工具有 fixture，但没有成体系的 pytest/CI；UE 构建和烟测仍主要靠手工脚本。



推荐开发方向


先做可复现基线

初始化/锁定 URLab 子模块，修复启动脚本的依赖检查；跑一次 scripts/airspark_ue.ps1 -Build 和最小 School batch smoke。没有这个基线，后续问题很难区分是代码缺陷还是环境缺陷。



清理文档事实源

以 ROADMAP.md + docs/wiki/AirSpark-Plugin-Architecture.md 为准，删除或修正旧的 stub/placeholder 文档。否则协作成本会持续上升。



收尾 P2 数据工厂

实现 max_retries_per_episode，把真实 image payload、frame_sensor_updates.jsonl、manifest summary、validator strict 模式跑成一个可重复的 3/10/100 episode 验收矩阵。



补齐导航核心

优先实现可用的 3D occupancy grid / clearance / A* 或 Theta* / smoothing / sweep validation。当前项目的论文/benchmark 价值主要取决于“任务可生成、路径可验证、失败可解释”。



进入 P3：SceneSpec 平台化

ROADMAP 里 SceneSpec 方向是对的：不要追求“一句话生成 UE 世界”，而是把外部输入统一转为结构化 SceneSpec，再生成地图、任务、约束、评估指标。见 ROADMAP.md (line 128)。



多 drone 前先拆全局状态

把 RuntimeProfile.override、collector context、capture queue、MissionDirector vehicle name 改成 per-vehicle/per-episode context，再谈并行采集。



数据产品化

LeRobot v3 真实 video/lowdim 导出、QA dashboard、coverage report、failure mining、regeneration planner 应成为下一阶段核心产物，而不是继续扩展临时 Python 脚本。



整体判断：仓库已经从“飞行仿真 demo”演进到“VLA 数据工厂雏形”，方向是清楚的；当前最大问题不是缺想法，而是 复现基线、文档一致性、规划器真实能力、批采集鲁棒性 还没完全收口。
